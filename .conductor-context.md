# Conductor V2 Improvement Context

## Codebase Summary

This is the Claude Code Conductor (C3) — a hierarchical multi-agent orchestration system in TypeScript. Current version: 0.1.6.

### Key Files
- `src/core/orchestrator.ts` (2214 lines) — Main loop: init → plan → conventions → execute → review + flow-trace → checkpoint
- `src/core/worker-manager.ts` (581 lines) — Spawns/monitors headless Claude Code workers via Agent SDK
- `src/core/state-manager.ts` (437 lines) — Persistent state with atomic write (temp+rename) pattern
- `src/mcp/tools.ts` (712 lines) — 11 MCP coordination tools for workers (claim_task, complete_task, contracts, decisions)
- `src/mcp/coordination-server.ts` — MCP server setup that registers the tools
- `src/utils/types.ts` — All TypeScript type definitions
- `src/utils/constants.ts` (217 lines) — Configuration constants, tool allowlists
- `src/worker-prompt.ts` (269 lines) — Worker system prompt builder
- `src/worker-personas.ts` (413 lines) — 7 specialized personas per task type
- `src/core/planner.ts` (664 lines) — Plan creation with STRIDE threat modeling
- `src/core/flow-tracer.ts` (549 lines) — Flow extraction and tracing
- `src/cli.ts` — CLI commands: start, status, resume, pause, log
- `src/utils/known-issues.ts` — Known issues registry across cycles
- `src/utils/semgrep-runner.ts` — Semgrep static analysis wrapper
- `src/core/codex-reviewer.ts` — Codex CLI integration for plan/code review

### What Already Exists (v0.1.6)

- Phase timing (PhaseDurations instrumented in orchestrator)
- Blast radius detection (computeBlastRadius)

## Improvements to Implement

### 1. Worker Resilience & Lifecycle
**Problem:** No wall-clock timeout per worker (only SDK maxTurns=100). No retry for failed tasks. No heartbeat detection.

**Implementation plan:**
- Create `src/utils/worker-resilience.ts` with:
  - `WorkerTimeoutTracker` class: tracks start time per worker, checks timeouts (default 30min)
  - `TaskRetryTracker` class: records failures, decides retry (max 2 retries), generates retry context for re-injection
  - `HeartbeatTracker` class: records worker activity from events, detects stalled workers (5min threshold)
- Integrate into `worker-manager.ts`:
  - Track timeouts when spawning workers
  - Record heartbeats from processWorkerEvent
- Integrate into orchestrator execution loop:
  - Check timeouts and stalled workers in the monitoring loop
  - Reset timed-out worker tasks to pending with retry context
  - After execution, check failed tasks and create retry tasks if attempts remain
- Add `retry_count` and `last_error` fields to Task type
- Add `DEFAULT_WORKER_TIMEOUT_MS` constant (30 * 60 * 1000)

### 2. Smart Task Scheduling
**Problem:** Tasks claimed in ID order (FIFO). No priority. Critical path tasks should go first.

**Implementation plan:**
- Create `src/utils/task-scheduler.ts` with:
  - `computeCriticalPathDepths(tasks)`: compute max downstream chain length per task
  - `scoreTask(task, depth)`: score = (depth * 10) + riskScore + typeScore
  - `rankClaimableTasks(allTasks)`: filter pending+unblocked, score, sort descending
- Modify `mcp/tools.ts` `handleGetTasks()`:
  - Add optional `ranked: true` parameter
  - When ranked=true, return tasks sorted by priority instead of by ID
- Update worker prompt to instruct workers to claim highest-priority task from get_tasks

### 3. Project Auto-Detection
**Problem:** Workers get generic guidance. No framework/test runner/linter detection.

**Implementation plan:**
- Create `src/utils/project-detector.ts` with:
  - `detectProject(projectDir)`: returns `ProjectProfile` (language, frameworks, testRunner, linters, ci, packageManager)
  - `formatProjectGuidance(profile)`: generates markdown guidance for workers
  - Detects: Node.js (Next/Express/NestJS/React/Vue), Python (FastAPI/Django), Rust, Go, Ruby (Rails)
  - Detects test runners: vitest/jest/mocha/pytest/cargo test/go test
  - Detects linters: eslint/prettier/biome/ruff/clippy
  - Detects CI: GitHub Actions, GitLab CI, CircleCI
- Integrate into orchestrator init phase (after directory creation, before planning)
- Pass detected profile into worker context for prompt injection

### 4. Structured Event Log
**Problem:** Logger writes plain text. No structured events. Hard to debug/analyze runs.

**Implementation plan:**
- Create `src/utils/event-log.ts` with:
  - `EventLog` class with JSONL append to `.conductor/events.jsonl`
  - Buffered writes (flush every 1s or on shutdown)
  - Convenience methods: phaseStart/End, workerSpawned/Completed/Failed/TimedOut, taskClaimed/Completed/Failed/Retried, reviewVerdict, usageWarning, schedulingDecision, projectDetected
  - `readAll()` and `getPhaseSummary()` for analysis
- Integrate into orchestrator: log phase transitions, worker lifecycle, task lifecycle
- Add to CLI status command: show event summary

**Implementation plan:**
