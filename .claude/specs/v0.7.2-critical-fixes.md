## Goals

Ship a narrow, security-focused patch that addresses the audit's actually-exploitable issues:

1. **CR-1** — Worker "read-only" restriction is advisory, not enforced. **10 read-only callsites** (sentinel, flow-tracer×2, planner question-gen, prompt-compactor, and 5 analyzer/extractor agents) can actually Write/Edit/Task despite their `allowedTools` suggesting otherwise.
2. **CR-2** — `CodexReviewer.runCodex` reads unbounded file contents into prompt; silent truncation can produce false-positive reviews.
3. **CR-3** — `CodexReviewer` never passes `--model` to codex CLI; security-gate review may run on a weaker model than execution workers.
4. **H-1** — Logger writes all messages (including any Bearer tokens, sk-ant-* keys) to disk plaintext with zero redaction.
5. **H-2** — `.conductor/` directories end up 0o755 because `fs.mkdir({ mode: 0o700 })` is filtered by umask and several call sites omit mode entirely.
6. **H-17** — Investigator empty-response fallback message is misleading ("plan has been updated"). Known bug in MEMORY.md.

We are **not** doing the full CR-1(b) (restricting execution workers via `tools`) in this patch — deferred to v0.8.0 because it may surface latent allowlist gaps. We are **not** doing the broader CR-4 test-coverage pass, but we will add tests for the modules we're already touching (`sanitize.ts`, `secure-fs.ts`, `logger.ts` redaction).

### Residual security risk: Bash write-via-shell

After this patch, read-only workers will have `Write`, `Edit`, `NotebookEdit`, and `Task` stripped via `disallowedTools`. The scope of remaining Bash access splits into two categories (see full callsite table above for line numbers):

**Bash still present (residual write-via-shell risk) — 7 sites:**

A sufficiently determined prompt injection could still write a file via `echo ... > path` through Bash at the Bash-present sites. This gap is:
- **Known and documented** — called out in the PR description and commit message for CR-1.
- **Not addressed here** because properly sandboxing Bash requires either (a) a `canUseTool` callback that parses and filters shell commands (brittle), or (b) an OS-level sandbox (codex exec has `--sandbox read-only` but the Agent SDK does not expose equivalent at worker spawn time).
- **Tracked for v0.8.0** as a follow-up alongside the broader CR-1(b) restriction of execution workers via `tools`.

This patch materially improves security (Write/Edit/NotebookEdit/Task stripped where previously the prompt was the only barrier) while accepting the Bash residual at the enumerated sites.

## Fix 1 — CR-1: Enforce read-only workers via `disallowedTools` (all read-only callsites)

**Finding**: SDK's `allowedTools` only auto-approves tools through the permission gate; it does not restrict which tools the model sees. With `permissionMode: "bypassPermissions"` (C3's default — see `sdk-timeout.ts:61`), `allowedTools` is redundant. Verified against `node_modules/@anthropic-ai/claude-agent-sdk/sdk.d.ts:1133-1167` — SDK provides `tools` (restrict available set) and `disallowedTools` (remove from context) for actual enforcement.

## Fix 2 — CR-2: Cap file-read prompt content + switch to stdin in `runCodex`

**Finding**: `src/core/codex-reviewer.ts:358-371` — `runCodex` reads all file contents, concatenates into `prompt + fileContents.join("")`, and passes the result as a single argv entry via `execFile`. Two problems:
1. No size guard — multi-MB diffs can silently truncate inside Codex's token limit.
2. **OS argv limit (E2BIG)** — `getconf ARG_MAX` on macOS is 1MB; many Linux distros are 128KB–2MB. Combined with env and other args, a large prompt can hit `E2BIG` before Codex even runs. This is a real, hard failure that silent truncation would mask.

### Approach (revised after Codex review)

**Switch to stdin for prompt delivery**, not argv. `codex exec` supports `-` as the PROMPT argument to read from stdin (verified via `codex exec --help`: *"If not provided as an argument (or if `-` is used), instructions are read from stdin."*). This eliminates argv limits entirely.

Then add a sane aggregate cap (2MB) purely to bound Node heap + Codex token cost, not for argv reasons.

**`src/utils/constants.ts`** — add:

```ts
// CodexReviewer prompt size guards (CR-2)
// Per-file cap: 100K UTF-16 code units. Large enough for most files but
// prevents one giant file from dominating the prompt.
export const MAX_CODEX_PROMPT_FILE_CHARS = 100_000;
// Aggregate **file-content** cap: 2 million UTF-16 code units (~2-4MB depending
// on encoding). This cap applies only to the concatenated file chunks appended
// to the prompt — it does NOT include the base prompt prefix or the summary
// marker. Total stdin payload will be `prompt + file-chunks + summary` and may
// slightly exceed the aggregate cap by prefix+summary length (typically <5KB).
// Bounds Node heap pressure and Codex token cost. The cap is a character
// count, not a byte count — simple to reason about, accepts ~2x overhead
// on multi-byte content, fine for its purpose (not a hard memory budget).
// Prompt is delivered via stdin (not argv) so OS ARG_MAX is not a factor.
export const MAX_CODEX_PROMPT_AGGREGATE_CHARS = 2_000_000;
// spawn output byte cap (parity with the 10MB execFile maxBuffer that's
// currently set at codex-reviewer.ts:397). Exceeded stdout → kill +
// CodexExecutionError("output_too_large").
export const MAX_CODEX_STDOUT_BYTES = 10 * 1024 * 1024;
```

**`src/core/codex-reviewer.ts:358-371`** — modify `runCodex`:

```ts
/**
 * Read at most `maxChars` UTF-16 code units from a file without allocating
 * the whole file in memory. Uses fs.open + bounded read. Returns:
 *   { content, fullByteLength, wasTruncated }
 * where wasTruncated is authoritatively true iff the file exceeded the cap
 * AND the returned content was actually sliced. Callers should not infer
 * truncation from byte/char mismatches — that's unreliable for multi-byte
 * content (a fully-read 50KB UTF-8 file has 50KB bytes but may have only
 * 12KB chars).
 */
async function readFilePrefix(p: string, maxChars: number): Promise<{
  content: string;
  fullByteLength: number;
  wasTruncated: boolean;
}> {
  const stat = await fs.stat(p);
  const fullByteLength = stat.size;
  // Read ~4× maxChars bytes to cover worst-case UTF-8 expansion, then slice
  // the decoded string to maxChars. Still bounded, no full-file allocation.
  const readBytes = Math.min(fullByteLength, maxChars * 4);
  const fh = await fs.open(p, "r");
  try {
    const buf = Buffer.alloc(readBytes);
    const { bytesRead } = await fh.read(buf, 0, readBytes, 0);
    const decoded = buf.subarray(0, bytesRead).toString("utf-8");
    const wasTruncated =
      bytesRead < fullByteLength || decoded.length > maxChars;
    const content = wasTruncated ? decoded.slice(0, maxChars) : decoded;
    return { content, fullByteLength, wasTruncated };
  } finally {
    await fh.close();
  }
}

private async runCodex(prompt: string, readPaths: string[]): Promise<string> {
  const fileContents: string[] = [];
  let aggregateChars = 0;
  let truncatedFiles = 0;
  let skippedAfterCap = 0;
  let capReached = false;

  for (const p of readPaths) {
    try {
      // Memory-safe: only read up to the per-file cap, not the whole file.
      const { content, fullByteLength, wasTruncated } = await readFilePrefix(
        p,
        MAX_CODEX_PROMPT_FILE_CHARS,
      );
      const basename = path.basename(p);

      if (capReached) {
        skippedAfterCap++;
        continue; // no per-file marker; single summary marker appended after loop
      }

      const remaining = MAX_CODEX_PROMPT_AGGREGATE_CHARS - aggregateChars;
      if (remaining <= 0) {
        capReached = true;
        skippedAfterCap++;
        continue;
      }

      // Per-file truncation marker only when readFilePrefix says so.
      let body = content;
      if (wasTruncated) {
        body = content + `\n\n[TRUNCATED — file is ${fullByteLength} bytes, showing first ${content.length} chars]`;
        truncatedFiles++;
      }

      const wrapperOpen = `\n\n## File: ${basename}\n\n\`\`\`\n`;
      const wrapperClose = "\n```";
      const fullChunk = wrapperOpen + body + wrapperClose;

      let chunk = fullChunk;
      if (chunk.length > remaining) {
        const marker = `\n\n[TRUNCATED_BY_AGGREGATE_CAP]`;
        chunk = fullChunk.slice(0, Math.max(0, remaining - marker.length)) + marker;
        truncatedFiles++;
        capReached = true; // further files are skipped-by-summary
      }

      fileContents.push(chunk);
      aggregateChars += chunk.length;
    } catch (err) {
      this.logger.warn(`Could not read file ${p}: ${String(err)}`);
    }
  }

  // Single summary marker for all files skipped due to aggregate cap —
  // does NOT contribute per-file overflow.
  if (skippedAfterCap > 0) {
    fileContents.push(
      `\n\n[${skippedAfterCap} additional file(s) skipped due to aggregate prompt cap of ${MAX_CODEX_PROMPT_AGGREGATE_CHARS} chars.]`
    );
  }

  if (truncatedFiles > 0 || skippedAfterCap > 0) {
    this.logger.warn(
      `Codex reviewer prompt: ${truncatedFiles} file(s) truncated, ${skippedAfterCap} file(s) skipped (aggregate cap ${MAX_CODEX_PROMPT_AGGREGATE_CHARS}). Review may be incomplete.`
    );
  }

  const fullPrompt = prompt + fileContents.join("");

  // CR-2: deliver prompt via stdin (not argv) to avoid OS ARG_MAX (E2BIG)
  // on large diffs. "-" tells codex exec to read PROMPT from stdin.
  const args: string[] = [
    "exec",
    "--full-auto",
    "--model", codexModel,          // CR-3
    "-C", this.projectDir,
    "-c", 'mcp_servers.coordinator.command="node"',
    // ... other -c args unchanged ...
    "-",                              // prompt from stdin
  ];

  // Use spawn so we can pipe to stdin; execFileAsync won't cover this cleanly.
  // Promisify a spawn + drain pattern. Retain REVIEW_TIMEOUT_MS and maxBuffer.
  const { stdout, stderr } = await spawnCodexWithStdin(args, fullPrompt, {
    timeout: REVIEW_TIMEOUT_MS,
    maxBuffer: 10 * 1024 * 1024,
    cwd: this.projectDir,
  });
  // ... rest of error handling unchanged
}
```

Helper `spawnCodexWithStdin` wraps `child_process.spawn` with these guarantees (parity with `execFileAsync`'s implicit behaviors):

- Writes `fullPrompt` to `proc.stdin` and closes it.
- Accumulates stdout/stderr with a **byte-count maxBuffer guard** (`spawn` has no built-in maxBuffer). Track `stdoutBytes += Buffer.byteLength(chunk)` and `stderrBytes += Buffer.byteLength(chunk)` in `data` handlers — use `Buffer.byteLength`, NOT string length, so multibyte content cannot bypass the threshold. On overflow:
  - Kill the child process (`proc.kill("SIGTERM")`, fall back to `SIGKILL` after grace).
  - Throw `CodexExecutionError("stdout exceeded 10MB...", "output_too_large")` — new error kind added to the `CodexExecutionReason` union.
  - **Handling in `withRetryOnInvalidResponse`** — `output_too_large` must be terminal in BOTH catch blocks (first attempt at `codex-reviewer.ts:258-266`, second attempt at `:294-302`). Retrying with the same over-large prompt will fail again, so no retry. But throwing would hard-stop the conductor at `orchestrator.ts:548` — treating it as a terminal `ERROR` verdict keeps orchestration resilient while still being visible in review output.

    Shared helper inside the class (avoids duplicated code across the two catches):

## Fix 4 — H-1: Logger secret redaction

**Finding**: `src/utils/logger.ts:93-97` writes every line directly to disk with no filtering. Any token/key in a log line (common in SDK error messages, auth-related debug) persists plaintext. Compounded by H-2 (world-readable dir).

### Approach

Add a module-level `redactSecrets()` function. Apply in `writeToFile()`.

**`src/utils/logger.ts`**:

```ts
/**
 * Redact likely secrets from a log line before writing to disk.
 * Not a substitute for not logging secrets — defense in depth.
 *
 * Patterns cover:
 *  - Anthropic API keys (sk-ant-*)
 *  - Authorization header values (Bearer *, Authorization: *)
 *  - JSON-ish secret fields ("token":"...", "api_key":"...", "password":"...")
 *  - URL-param-ish secret fields (token=..., api_key=..., password=...)
 *  - Long hex strings (likely session tokens / SHA-512 / HMACs)
 */
const SECRET_PATTERNS: Array<[RegExp, string | ((m: string) => string)]> = [
  [/sk-ant-[A-Za-z0-9_-]{20,}/g, "sk-ant-***REDACTED***"],
  [/Authorization\s*[:=]\s*[A-Za-z0-9._~+/=-]+/gi, "Authorization: ***REDACTED***"],
  [/Bearer\s+[A-Za-z0-9._~+/=-]{10,}/gi, "Bearer ***REDACTED***"],
  // JSON-style: "token":"abc123", "api_key": "..." (case-insensitive keys)
  [/"(api[_-]?key|token|secret|password|passwd|auth)"\s*:\s*"[^"]{6,}"/gi,
   (m: string) => m.replace(/:\s*"[^"]+"/, ': "***REDACTED***"')],
  // URL/config-style: token=abc123, api_key=..., password=...
  [/\b(api[_-]?key|token|secret|password|passwd)\s*[:=]\s*[A-Za-z0-9._~+/=-]{8,}/gi,
   (m: string) => m.replace(/([:=]\s*)[A-Za-z0-9._~+/=-]{8,}/, "$1***REDACTED***")],
  // Long hex strings (40+ chars) — SHA-1s, HMACs, session tokens. May match
  // commit SHAs but those aren't secret; we accept the collateral damage.
  [/\b[a-f0-9]{40,}\b/g, "***HEX_REDACTED***"],
];

export function redactSecrets(line: string): string {
  let out = line;
  for (const [re, replacement] of SECRET_PATTERNS) {
    out = typeof replacement === "string"
      ? out.replace(re, replacement)
      : out.replace(re, replacement as (s: string) => string);
  }
  return out;
}

// In writeToFile:
private writeToFile(line: string): void {
  if (this.closed) return;
  this.logStream.write(redactSecrets(line) + "\n");
}
```

Console output (line 54/62/70/82) remains unredacted so live interactive debugging is unaffected. Redaction only applies to on-disk persistence. This tradeoff is intentional — if the user objects, we can extend to console too.

**Tests**: `src/utils/logger.test.ts` — covers each pattern, ensures benign text is not clobbered, asserts hex-redaction doesn't destroy long SHAs (accept collateral damage on that pattern — comment it).

import fsSync from "node:fs";

export function mkdirSecureSync(dirPath: string, options?: { recursive?: boolean; mode?: number }): void {
  const recursive = options?.recursive ?? false;
  const mode = options?.mode ?? SECURE_DIR_MODE;
  fsSync.mkdirSync(dirPath, { recursive, mode });
  try {
    fsSync.chmodSync(dirPath, mode);
  } catch {
    // ignore — may not own dir
  }
}
```
