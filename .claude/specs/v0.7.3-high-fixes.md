## Fix 1 ‚Äî H-4: `Promise.all` ‚Üí `Promise.allSettled` for parallel review phases

**Finding**: `src/core/orchestrator.ts:436` runs code-review, flow-tracing, and design-spec-update in parallel via `Promise.all`. If any leg rejects, sibling results are silently lost. Flow-trace findings feed the checkpoint security gate ‚Äî losing them can let a cycle pass despite unseen critical issues.

### Current code

```ts
// orchestrator.ts:436-440
const [approved, flowReport, _designSpecUpdate] = await Promise.all([
  this.review().then((r) => { phaseDurations.code_review_ms = Date.now() - reviewStart; return r; }),
  this.flowReview(cycleNum).then((r) => { phaseDurations.flow_tracing_ms = Date.now() - flowStart; return r; }),
  this.updateDesignSpecIfNeeded(cycleNum),
]);
```

### Approach

Switch to `Promise.allSettled`. Treat the three legs differently based on whether they gate the security checkpoint:

- **Code review** (`review()`): security gate ‚Äî if it throws, treat as review failure (`approved = false`) and log loudly. Do not advance to checkpoint silently; let the checkpoint's own branch (`if (!approved)` around line 471) force another cycle.
- **Flow tracing** (`flowReview()`): security gate ‚Äî if it throws, record a synthetic high-severity finding to force another cycle. This mirrors what actually-finding-an-issue would have done; prevents checkpoint from advancing on a silently-dropped gate.
- **Design spec update** (`updateDesignSpecIfNeeded()`): non-gating ‚Äî if it throws, log a warning and continue. Design spec staleness is a convenience, not a security invariant.

### Files & changes

**`src/core/orchestrator.ts:436-440`** ‚Äî replace the block:

## Fix 2 ‚Äî H-5: state.json backup + fallback-on-load recovery

**Finding**: `src/core/state-manager.ts:110-124` ‚Äî `load()` throws on malformed state.json with no recovery path. No `.bak` is ever written. A torn write or partial corruption renders `conduct resume` unusable; the known-issues registry (cross-cycle security memory) is lost.

### Approach

Write a `.bak` copy at every successful save. On load failure, attempt `.bak`; log a warning and surface the path. Keep the original `.bak` so if recovery succeeds, the user can inspect what differed.

### Files & changes

**`src/core/state-manager.ts`** ‚Äî two changes:

**Prerequisite**: `StateManager` does not currently have a logger dependency (verified ‚Äî no `this.logger` references in `state-manager.ts`). Do NOT add one here ‚Äî use a local warn sink (`process.stderr.write`) keyed by `STATE_MANAGER_WARN` prefix, so CLI output stays visible without introducing a new dependency.

```ts
// Near top of state-manager.ts, below imports:
function warnStateManager(msg: string): void {
  process.stderr.write(`[state-manager] ${msg}\n`);
}
```

1. In `save()` around line 160 (after atomic rename succeeds): copy the live state.json to `.bak`. Use a separate atomic write (`state.json.bak.tmp` ‚Üí rename). Do this **after** the main rename so a crash mid-save never overwrites a good `.bak` with a bad state.

```ts
// After: await fs.rename(tmpPath, statePath);
// Update .bak only on successful main save
try {
  const backupPath = statePath + ".bak";
  const backupTmp = backupPath + ".tmp";
  await fs.writeFile(backupTmp, content, { encoding: "utf-8", mode: 0o600 });
  // fsync is NOT required on .bak ‚Äî it's a redundant copy; if we lose it,
  // main state.json is still authoritative. Atomic rename is enough.
  await fs.rename(backupTmp, backupPath);
} catch (err) {
  // Non-fatal: .bak write failure should not block the save.
  warnStateManager(`Failed to update state.json.bak: ${String(err)}`);
}
```

2. In `load()` around line 110-124: try primary, on failure try `.bak`. **Post-recovery save failure is non-fatal** ‚Äî if we successfully recovered state from `.bak` but the follow-up `save()` fails, we keep the recovered state in-memory and surface a warning. The caller can still operate; next successful save will re-establish `.bak`.

```ts
async load(): Promise<OrchestratorState> {
  const statePath = getStatePath(this.projectDir);
  try {
    const raw = await fs.readFile(statePath, "utf-8");
    const result = validateStateJsonLenient(raw);
    if (!result.valid) {
      throw new Error(
        `State file validation failed (${statePath}):\n${result.errors.map((e) => `  - ${e}`).join("\n")}`,
      );
    }
    this.state = result.state as OrchestratorState;
    return this.state;
  } catch (primaryErr) {
    const backupPath = statePath + ".bak";
    let backupState: OrchestratorState;
    try {
      const backupRaw = await fs.readFile(backupPath, "utf-8");
      const backupResult = validateStateJsonLenient(backupRaw);
      if (!backupResult.valid) {
        throw new Error(
          `state.json.bak also failed validation:\n${backupResult.errors.map((e) => `  - ${e}`).join("\n")}`,
        );
      }
      backupState = backupResult.state as OrchestratorState;
    } catch (backupErr) {
      throw new Error(
        `State load failed and backup recovery also failed.\n` +
        `Primary (${statePath}): ${String(primaryErr)}\n` +
        `Backup (${backupPath}): ${String(backupErr)}`,
      );
    }

    // Preserve the corrupt primary for forensics (unique timestamped name).
    // Only report preservation in the warning if the rename actually succeeded ‚Äî
    // otherwise the operator gets a misleading path that doesn't exist.
    const corruptTs = Date.now();
    const corruptPath = `${statePath}.corrupt-${corruptTs}`;
    let preserved = false;
    try {
      await fs.rename(statePath, corruptPath);
      preserved = true;
    } catch (renameErr) {
      // Non-fatal ‚Äî primary may already be gone / unreadable
      warnStateManager(
        `Could not preserve corrupt state.json for forensics: ${String(renameErr)}`,
      );
    }
    warnStateManager(
      `state.json corrupt (${String(primaryErr)}); recovered from ${backupPath}.` +
      (preserved ? ` Original preserved at ${corruptPath}.` : ""),
    );

    this.state = backupState;
    // Best-effort: persist recovered state so next load finds a valid primary.
    // A failure here is non-fatal ‚Äî we already have valid state in-memory.
    try {
      await this.save();
    } catch (saveErr) {
      warnStateManager(
        `Recovered state loaded from .bak but post-recovery save failed: ${String(saveErr)}. ` +
        `State is valid in memory; next successful save will re-establish state.json.`,
      );
    }
    return this.state;
  }
}
```

Note on locking: `load()` does NOT hold a `proper-lockfile` lock, and `save()` acquires/releases its own lock. The `this.save()` call from inside load's recovery branch is safe ‚Äî no reentrancy (verified: `state-manager.ts:148-170` shows independent lock lifecycle).

## Fix 3 ‚Äî H-6: fsync before rename in atomic state write

**Finding**: `src/core/state-manager.ts:157-160` ‚Äî `fs.writeFile(tmpPath, ...)` is followed directly by `fs.rename(tmpPath, statePath)` with no `fh.sync()` between. On power-loss (not just process crash), the rename can complete while tmp data is still in page cache; after reboot, state.json can be zero-length or contain partial garbage. Zero `fsync` calls exist in the codebase.

### Approach

Replace `fs.writeFile(tmpPath, content, ...)` with an explicit `fs.open` + `fh.writeFile` + `fh.sync` + `fh.close` sequence before the rename. Same `.bak` update in Fix 2 does NOT need fsync ‚Äî it's a redundant copy.

### Files & changes

**`src/core/state-manager.ts:157`** ‚Äî replace:

```ts
// Was:
await fs.writeFile(tmpPath, content, { encoding: "utf-8", mode: 0o600 });
// Atomic rename
await fs.rename(tmpPath, statePath);

// Becomes (H-6):
const tmpFh = await fs.open(tmpPath, "w", 0o600);
try {
  await tmpFh.writeFile(content, { encoding: "utf-8" });
  await tmpFh.sync(); // flush file data to disk before rename
} finally {
  await tmpFh.close();
}
await fs.rename(tmpPath, statePath);

// Parent-dir fsync for full power-loss durability (per Codex review round 3).
// Without this, the rename can be lost on power loss even though the file data
// was fsynced. Wrapped in try/catch because some filesystems (network mounts,
// certain fuse FSs) do not support directory fsync and return EINVAL.
try {
  const dirFh = await fs.open(path.dirname(statePath), "r");
  try {
    await dirFh.sync();
  } finally {
    await dirFh.close();
  }
} catch (err) {
  // Non-fatal ‚Äî file data was synced, and most failures here (e.g. EINVAL
  // on unsupported filesystems) don't indicate real durability loss.
  warnStateManager(`Parent-dir fsync after state.json rename failed: ${String(err)}`);
}
```

### Tests

New in `src/core/state-manager.test.ts`:

1. **fsync called before rename** ‚Äî mock `fs.promises.open` to return a FileHandle whose `sync` method is tracked. Call `save()`, assert `sync` was called exactly once and before `fs.rename`.
2. **File closed on sync failure** ‚Äî mock `sync` to throw. Assert `close` is still called (finally block), and the error propagates.

## Fix 5 ‚Äî H-8: `composeRoleConfig` field-level merge

**Finding**: `src/cli.ts:272-278` ‚Äî `merged = { ...merged, ...fileResult.roles }` and `merged = { ...merged, ...legacyExpanded }` are shallow object spreads. When a later layer supplies a partial RoleModelSpec (tier only), it fully replaces the earlier layer's value, losing the inherited effort. Compounds H-7.

### Approach

Define a `mergeRoleSpec(base, patch)` helper (or reuse `applyRolePatches` shape) that field-merges per role. Apply it at every composition layer.

### Files & changes

**`src/utils/models-config.ts`** ‚Äî add helper next to `DEFAULT_ROLE_CONFIG`:

```ts
/**
 * Field-level merge of per-role spec maps. For each role, the patch's fields
 * are spread over the base's fields ‚Äî so a tier-only patch does NOT wipe the
 * base's effort. This is the intended behavior for every layer of
 * composeRoleConfig (state seed ‚Üí file ‚Üí legacy expansion ‚Üí CLI patches).
 */
export function mergeRoleMaps(
  base: Partial<Record<AgentRole, RoleModelSpec>>,
  patch: Partial<Record<AgentRole, Partial<RoleModelSpec>>> | undefined,
): Partial<Record<AgentRole, RoleModelSpec>> {
  if (!patch) return base;
  const result: Partial<Record<AgentRole, RoleModelSpec>> = { ...base };
  for (const role of Object.keys(patch) as AgentRole[]) {
    const p = patch[role];
    if (!p) continue;
    const b = result[role];
    if (!b) {
      // Completely new entry ‚Äî require both fields. Missing fields fall back
      // to DEFAULT_ROLE_CONFIG (H-7 parity).
      result[role] = {
        tier: p.tier ?? DEFAULT_ROLE_CONFIG[role].tier,
        effort: p.effort ?? DEFAULT_ROLE_CONFIG[role].effort,
      };
    } else {
      result[role] = {
        tier: p.tier ?? b.tier,
        effort: p.effort ?? b.effort,
      };
    }
  }
  return result;
}
```

```ts
import { WORKER_ALLOWED_TOOLS } from "./constants.js";
import { TASK_TYPE_LITERALS } from "./types.js";

/**
 * Host-side verification of LLM-extracted rules. Returns true if the document
 * passes sanity checks, false if drift was detected. Caller falls back to
 * FALLBACK_TEMPLATE on false.
 *
 * Strict-when-triggered policy: if the document mentions the concept, every
 * concrete value must appear. Silent omission ‚Üí discard. This is the whole
 * point of host-side verification ‚Äî "tolerate some missing" defeats it.
 */
function verifyExtractedRules(rules: string, warn: (msg: string) => void): boolean {
  const mentionsTaskTypes = /task[_\s-]?type/i.test(rules);
  if (mentionsTaskTypes) {
    const missing = TASK_TYPE_LITERALS.filter((t) => !rules.includes(t));
    if (missing.length > 0) {
      warn(
        `Rules document mentions task types but omits: ${missing.join(", ")}. ` +
        `Discarding extracted rules and falling back to template.`,
      );
      return false;
    }
  }

  const mentionsAllowedTools = /allowed[_\s-]?tools?|WORKER_ALLOWED_TOOLS/i.test(rules);
  if (mentionsAllowedTools) {
    // Exclude MCP tools (`mcp__*`) ‚Äî the agent is typically instructed to use
    // short names or descriptive wording for those. Strict-when-triggered
    // still applies to the non-MCP concrete tool names.
    const expected = WORKER_ALLOWED_TOOLS.filter((t) => !t.startsWith("mcp__"));
    const missing = expected.filter((t) => !rules.includes(t));
    if (missing.length > 0) {
      warn(
        `Rules document mentions allowed-tools but omits: ${missing.join(", ")}. ` +
        `Discarding extracted rules and falling back to template.`,
      );
      return false;
    }
  }

  return true;
}
```

```ts
 * sibling, fsync, then rename. Rename is atomic on POSIX within a filesystem.

 * Mode defaults to 0o600. Pass `{ fsync: false }` to skip the durability
 * flush ‚Äî rarely correct, but allowed for non-critical files where the small
 * perf win matters.
 */
export async function writeJsonAtomic(
  destPath: string,
  content: string,
  options?: { mode?: number; fsync?: boolean },
): Promise<void> {
  const mode = options?.mode ?? 0o600;
  const wantFsync = options?.fsync !== false;

  } finally {

  }
  await fs.rename(tmpPath, destPath);
}
```

## Fix 15 ‚Äî H-19: Bound `read_updates` message history

**Finding**: `src/mcp/tools.ts:230-275` ‚Äî reads all `.jsonl` files in the messages dir, parses every line, filters in-memory. The `since` parameter filters after loading. Long-running conductors accumulate unbounded history; workers calling `read_updates` after many cycles load megabytes for zero new messages.

### Approach (revised twice after Codex review)

This fix is a **partial mitigation**, not a full resolution. The audit framing (H-19) is "unbounded message history" ‚Äî a full fix would require either streaming reads or a mandatory `since` parameter. Both are invasive and out of scope for a patch release. What this fix delivers:

1. **Pre-filter by mtime** (invisible, no contract change): if `since` is supplied and a file's mtime is older than `since`, skip parsing it entirely. This is the realistic worker pattern ‚Äî workers track a high-water mark and pass it as `since` ‚Äî so the common case sees a major memory win. **Not a fix for callers that omit `since`** or for cases where one active `*.jsonl` is very large; those still read everything.

2. **Per-call message cap** (opt-in, `limit` defaults to no cap): callers who want a hard cap pass `limit: N`; otherwise they receive every matching message (contract preserved). Internal max is `MAX_READ_UPDATES_HARD_CAP = 10000` ‚Äî a safety rail for `limit: Infinity`, not a default applied to unsuspecting consumers.

The remaining unbounded-read risk for no-`since` consumers and huge-active-file cases is **documented, not fixed**. A v0.8.0 item should address it via one of: (a) stream + filter per-line without loading whole files, (b) require `since` at the schema level, or (c) rotate the active JSONL file on a size threshold. All three are larger changes than what belongs in this patch.

### Files & changes

**`src/mcp/tools.ts:226-275`**:

```ts
export interface ReadUpdatesInput {
  since?: string;
  /**
   * Optional cap on returned messages. When omitted, all matching messages
   * are returned (current behavior). When supplied, returns the `limit` most
   * recent messages by timestamp. Clamped to MAX_READ_UPDATES_HARD_CAP.
   */
  limit?: number;
}

// Hard safety rail for misbehaving callers. Not a default.
const MAX_READ_UPDATES_HARD_CAP = 10_000;

export async function handleReadUpdates(
  input: ReadUpdatesInput
): Promise<Message[]> {
  const dir = messagesDir();
  await ensureDir(dir);

  const sessionId = getSessionId();
  const rawSinceTs = input.since ? new Date(input.since).getTime() : 0;
  const sinceTs = Number.isNaN(rawSinceTs) ? 0 : rawSinceTs;
  const limit = typeof input.limit === "number"
    ? Math.max(1, Math.min(input.limit, MAX_READ_UPDATES_HARD_CAP))
    : null; // no cap when unspecified ‚Äî preserves pre-H-19 contract

  let dirents: string[];
  try {
    dirents = await fs.readdir(dir);
  } catch {
    return [];
  }

  const jsonlFiles = dirents.filter((f) => f.endsWith(".jsonl"));
  const allMessages: Message[] = [];

  for (const file of jsonlFiles) {
    const filePath = path.join(dir, file);
    // H-19: mtime pre-filter. If `since` is set and the file hasn't been
    // modified since then, no message inside it can be newer than `since`.
    // Invisible optimization ‚Äî doesn't change output vs. pre-H-19.
    if (sinceTs > 0) {
      try {
        const stat = await fs.stat(filePath);
        if (stat.mtimeMs <= sinceTs) continue;
      } catch {
        // Stat failure ‚Äî fall through to regular read
      }
    }
    const messages = await readJsonlFile<Message>(filePath);
    allMessages.push(...messages);
  }

  const filtered = allMessages.filter((msg) => {
    const msgTs = new Date(msg.timestamp).getTime();
    if (msgTs <= sinceTs) return false;
    if (msg.to && msg.to !== sessionId) return false;
    return true;
  });

  // Sort ascending by timestamp (existing contract).
  filtered.sort((a, b) => new Date(a.timestamp).getTime() - new Date(b.timestamp).getTime());

  if (limit === null) return filtered;

  // Caller asked for a limit ‚Äî return the `limit` MOST RECENT messages,
  // preserving ascending order within the returned slice.
  return filtered.slice(Math.max(0, filtered.length - limit));
}
```

1. **H-4 synthetic shape**: correct type is `FlowTracingReport` (not `FlowTraceReport`). `FlowFinding` requires `flow_id`, `actor`, `cross_boundary`, `file_path: string`. Synthetic report fills all required summary fields (`total`, `cross_boundary_count`, etc.) and uses visible-as-synthetic tokens (`<flow-tracing-infrastructure>`, `flow-tracing-failure-cycle-N`) so downstream `createFixTasksFromFindings` produces a clearly-identifiable fix task.
2. **H-4 per-leg timing**: preserve current per-promise `.then(...)` timing via chained handlers on each leg before `Promise.allSettled` ‚Äî don't stamp all durations from a single post-`allSettled` moment.
3. **H-5 logger**: `StateManager` has no logger dependency. Use a local `warnStateManager(msg)` helper that writes to `process.stderr`. Do not add a logger constructor param in this patch.
4. **H-5 post-recovery save failure**: non-fatal. Keep recovered state in-memory, surface warning, let next successful save re-establish state.json. No reentrancy issue (verified: load doesn't hold a lock, save acquires/releases independently).
5. **H-10 taskTypeHint**: `spawnWorkerForRetry` accepts `taskTypeHint` so retries keep role-specific model selection. Orchestrator passes `retryTask.task_type`. Interface `ExecutionWorkerManager.spawnWorkerForRetry` updated; Codex side accepts and ignores for parity.
6. **H-12 strict-when-triggered**: host-side verification has zero tolerance for missing literals once the document mentions the concept. TaskType literals live in a runtime tuple with a `satisfies readonly TaskType[]` + type-level exhaustiveness assertion, so adding a TaskType errors the build until the tuple is updated.
7. **H-15 concurrency plumbing**: CodexWorkerManager has zero concurrency references today. Added constructor param, updated orchestrator call site, and all test-utils that construct the manager.
8. **H-19 no default cap**: `limit` is opt-in. Default returns all matching messages (preserves existing contract). Hard safety rail at 10k only if caller supplies explicit `limit`. Primary memory bound is the mtime pre-filter, which doesn't change semantics.

## Decisions (resolved pre-review)

1. **v0.7.3 not v0.8.0**: all fixes, no features. Auto-archival stays for v0.8.0.
2. **Skip H-3**: SDK `tools` allowlist for execution workers deferred to its own v0.8.0 patch.
3. **Skip H-20**: 47 source-level test migration is rolling work, not release-blocking.
4. **H-9 uses planner role**: investigators are cross-cutting analysis agents; planner's model+effort is the right match. If a dedicated `investigator` role is added in v0.8.0, rewire both sites.
5. **H-4 handles each leg by gating behavior**: code-review and flow-trace are security gates; design-spec is not. Independent handlers, no all-or-nothing.
6. **H-5 preserves corrupt primary**: renamed to `.corrupt-<ts>` for forensics. User can inspect what went wrong without the recovery step clobbering evidence.
7. **H-12 is warn-and-fallback, not hard-fail**: the rules doc is injected into every worker prompt. Hard-failing because extraction drifted would block all runs ‚Äî the FALLBACK_TEMPLATE is the correct degraded state.
8. **H-14 introduces `writeJsonAtomic`**: shared helper in secure-fs.ts. Only used at escalation sites in this patch. Other non-atomic writers (tasks, contracts, decisions) stay as-is ‚Äî broader atomicity sweep is its own work.
9. **H-15 gates on `concurrency === 1`**, not `=== 1`-or-single-in-flight: simpler and correct. If we later want finer-grained gating, the comment points to where `--thread-id` support would enable it.
10. **H-19 adds `limit`, not `offset` pagination**: typical consumer pattern is "give me recent updates," not paginated scroll. `since`-based catch-up already exists; `limit` is the complementary bound.
