## Fix 1 — N-1: Sanitize `correctivePrompt` on the Codex side

**Finding**: `src/core/codex-worker-manager.ts` injects `correctivePrompt` (user-ish error text) into Codex CLI invocations at 3 sites without running it through `sanitizePromptSection()`. The Claude side was fixed in v0.7.4 A-6 (`worker-manager.ts:934`). Defense-in-depth gap — `sanitizeErrorForPrompt` at record time (worker-resilience.ts) pre-sanitizes, but the template framing around the corrective prompt in CodexWorkerManager is itself unsanitized.

### Sites (verified 2026-04-19)

`spawnWorkerForRetry` (`codex-worker-manager.ts:472`) has **three** downstream paths depending on the H-15 concurrency gate + thread-ID availability:

- **Path A — resume (concurrency=1 + preserved thread ID)**: line 500 assigns `resumePrompt = correctivePrompt ?? "default"`. Flows through `runCodexSessionWithResume` → `buildResumeArgs` → line 1094 where it's appended as a CLI arg. **Uses correctivePrompt.**
- **Path B — fresh-start fallback inside the resume branch**: line 720 `if (!resumeArgs)` triggers at 723 with `freshPrompt = \`${buildWorkerPrompt(sessionId)}\n\n## Retry Context\n\n... ${correctivePrompt}\``. **Uses correctivePrompt.**
- **Path C — fresh-start from the concurrency>1 gate at line 517**: calls `runCodexSession(sessionId, handle, this.buildWorkerPrompt(sessionId), ...)` with NO corrective context. **Drops correctivePrompt entirely.** Codex plan-review round 1 caught this.

Path C is a pre-existing gap: users running with concurrency>1 lose the retry corrective context because H-15's safety gate falls through to the plain `buildWorkerPrompt`. This isn't a v0.7.5-introduced regression — it was there since v0.7.3 H-15 — but we should fix it alongside the sanitization work since it's the same code shape.

### Approach (revised after round-1 feedback)

Extract a small pure helper that centralizes:
1. Sanitization via `sanitizePromptSection`
2. The "## Retry Context" preamble
3. The default fallback text when no correctivePrompt is provided

Then use the helper at all three paths. Benefits:
- One sanitization call site means no missed paths.
- The helper is pure, so testing is trivial (no mocking the Codex spawn).
- Path C gets the corrective context it was silently dropping.

```ts
// codex-worker-manager.ts — add import
import { sanitizePromptSection } from "../utils/sanitize.js";

// Add a private pure helper (near buildWorkerPrompt or at file end):
/**
 * Build the text to append as "retry context" on a Codex worker retry.
 * Handles sanitization (v0.7.5 N-1), default-fallback text, and the
 * "## Retry Context" preamble. Returns just the appendable string; caller
 * concatenates with the base prompt or CLI args as appropriate.
 */
private buildCorrectiveRetryText(correctivePrompt?: string): string {
  if (!correctivePrompt) {
    return "Continue working on this task. The previous attempt did not complete successfully.";
  }
  return `This is a retry of a previously failed task.\n\n${sanitizePromptSection(correctivePrompt)}`;
}
```

## Fix 3a — A-R2-prereq: Harden `writeJsonAtomic` tmp cleanup on failure

**Finding (Codex plan-review round 4)**: `writeJsonAtomic` in `src/utils/secure-fs.ts:200` does not delete the `.tmp` sibling if the write, fsync, close, or rename fails. The current `design-spec-updater.ts` manual implementation *does* unlink the tmp in a `finally`. Migrating A-R2 to `writeJsonAtomic` without hardening the helper would silently regress tmp-file cleanup on failure — leading to orphan `.tmp` files accumulating in `.conductor/` over time on systems that hit occasional write errors.

Apply the fix systemically in the helper rather than repeating caller-side cleanup at every site. v0.7.4 A-1 migrated 9 sites to `writeJsonAtomic`; those all inherit the same latent-leak bug today and benefit from this hardening.

### Approach

Refactor `writeJsonAtomic` to wrap the open→write→sync→close→rename sequence in try/finally. In the catch branch, attempt to unlink the tmp file; swallow the unlink error (tmp might not exist if `fs.open` itself failed).

```ts
// src/utils/secure-fs.ts:200 (approximate — adapt to actual signature)
export async function writeJsonAtomic(
  destPath: string,
  content: string,
  options?: { mode?: number; fsync?: boolean },
): Promise<void> {
  const mode = options?.mode ?? 0o600;
  const shouldFsync = options?.fsync !== false;
  const tmpPath = destPath + ".tmp";

  let renamed = false;
  try {
    const fh = await fs.open(tmpPath, "w", mode);
    try {
      await fh.writeFile(content, { encoding: "utf-8" });
      if (shouldFsync) await fh.sync();
    } finally {
      await fh.close();
    }
    await fs.rename(tmpPath, destPath);
    renamed = true;
    // post-rename chmod (unchanged)
    await fs.chmod(destPath, mode);
  } finally {
    // If we never successfully renamed, clean up the tmp file.
    // A successful rename consumes tmp, so skip in that case.
    if (!renamed) {
      try { await fs.unlink(tmpPath); } catch { /* tmp may not exist; swallow */ }
    }
  }
}
```

### Tests

Add 2 tests in `src/utils/secure-fs.test.ts` (existing `writeJsonAtomic` describe block from v0.7.4 T-5):

1. **Tmp cleaned up when write throws** — mock `FileHandle.writeFile` to throw; call `writeJsonAtomic`; assert `.tmp` does not exist afterward, and `destPath` unchanged (pre-existing file OR missing if it didn't exist).
2. **Tmp cleaned up when rename throws** — mock `fs.rename` to throw; call `writeJsonAtomic`; assert `.tmp` does not exist afterward.
