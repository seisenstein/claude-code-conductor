**`writeJsonAtomic` helper semantics** (verified at `secure-fs.ts:200-224`): content-agnostic string writer. Accepts any string. Uses `fs.open("w")` + `fh.writeFile(content)` + optional `fh.sync()` + `fs.rename()` + post-rename `chmod()`. Works identically for JSON, JSONL, and plain text тАФ no content awareness. All 9 A-1 sites write strings, so the helper is usable everywhere.

**Caveat for site #8 (`known-issues.ts:49`)**: the current code uses `writeFileSecure`, which is atomic in a different sense (non-atomic re filesystem crash, but mode-safe). Migration still applies тАФ `writeJsonAtomic` supersedes `writeFileSecure` for this use case (both target mode 0o600, but only the atomic variant survives crash).

**Caveat for site #5 (`state-manager.ts:383`)**: existing code writes `JSON.stringify(task, null, 2) + "\n"` with a trailing newline. Preserve the trailing newline in the migrated `writeJsonAtomic(taskPath, content)` call. The test at `orchestrator.test.ts:538` reads task files and verifies parsed structure but not trailing newline тАФ still, preserve it to minimize diff.

**validation.ts тАФ add `validateIdentifier`** (near `validateFileName`, ~line 100):

```ts
/**
 * Validates an identifier that becomes part of a filesystem path.
 * Stricter than validateFileName: rejects all path separators.
 *
 * Use for: task_id, dependency task_id, session_id, contract_id.
 * NOT for: files_changed (intentional relative paths).
 */
export function validateIdentifier(id: string): FileNameValidationResult {
  const base = validateFileName(id);
  if (!base.valid) return base;
  if (id.includes("/") || id.includes("\\") || id.includes(":")) {
    return {
      valid: false,
      reason: "Identifier cannot contain path separators (/, \\, :)",
    };
  }
  return { valid: true };
}
```

Note: `validateFileName` already rejects `\\`, so the backslash check is redundant but harmless (and makes the new function self-contained w.r.t. the `:` addition).
