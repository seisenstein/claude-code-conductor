# v0.7.6 — Auto-Archival Implementation Plan

**Target spec:** `.claude/specs/v0.7.6-auto-archival.md`
**Target version:** `0.7.6` (bump from `0.7.5`)
**Status:** Draft — awaiting Codex plan review

---

## 0. Scope Recap

We are adding auto-archival of conductor run artifacts under `.conductor/archive/<slug>-[FAILED-]<YYYYMMDD-HHMMSS>/`, plus a new `conduct archive` CLI subcommand group (`list`, `inspect`, `prune`, and default-action manual archive). Runs get archived automatically on terminal status (completed/failed) and on `conduct start` when a stale terminal state is detected in `.conductor/`. Paused/escalated runs are never archived.

The implementation strategy is sequenced so that unit-testable primitives land first and wiring into CLI + orchestrator comes last.

---

## 1. Key findings from codebase investigation

1. `src/utils/constants.ts` has stable helpers (`getOrchestratorDir`, `getStatePath`, etc.) and path constants. All new archive helpers slot in there.
2. `src/utils/secure-fs.ts` already provides `mkdirSecure` (0o700, umask-defeating) and `writeJsonAtomic` (tmp + fsync + rename + chmod 0o600). We reuse them — no custom atomic helpers.
3. `src/cli.ts` is the source of truth for the `.command("start")` / `.command("resume")` lifecycle. Lock acquisition flow at `cli.ts:707-715, 1167-1175` must be preserved — we hook archival between `acquireProcessLock` and `new Orchestrator(options)`.
4. `src/core/orchestrator.ts`:
   - `complete()` at line 2370 sets status `completed` then does cleanup. Hook archival at the END (after the console summary) so users still see the summary.
   - `run()` catch at line 675 sets status `failed`. We hook archival AFTER `setStatus("failed")` but BEFORE the `throw err` (still inside the try/catch of that scope, wrapped in its own try/catch — archival failure must never mask the original error).
5. `src/core/state-manager.ts` uses `StateManager.save()` which writes `state.json` and `state.json.bak`. Both must move into the archive. The `.bak` file is NOT in the spec's FILES_TO_ARCHIVE but IS part of every real run — we add it to the archive list.
6. `src/utils/validation.ts` has `validateIdentifier` (rejects slash/backslash/colon and URL-encoded forms). We reuse for the `inspect <slug>` and `prune` CLI arguments.
7. `commands/conduct.md` mentions the lifecycle but will need a one-paragraph note about stale-state auto-archival + `conduct archive list`.
8. `src/utils/gitignore.ts` already gitignores `.conductor/`, so archive dirs don't pollute VCS.
9. An EXISTING `.conductor/archive/` dir has two legacy archives (`v0.7.1-audit-20260418`, `v0.7.3-audit-20260418`) with NO `_archive-meta.json`. We must NOT corrupt them and `listArchives` must present them gracefully.
10. `package.json` version is `0.7.5`; `src/cli.ts` `.version("0.7.5")`. Both must bump.
11. Process lock details: `acquireProcessLock` in `cli.ts:418-522` creates `conductor.lock` and writes `conductor.lock.info`. proper-lockfile uses a `conductor.lock/` directory as lock state. These are currently held by the CLI process — **archival must NOT touch them**. They're transient; normal `releaseLock()` handles cleanup.

### 2.1 `src/core/archiver.ts`

Pure module with **no side-effects at import time**. Exports:

```ts
export interface ArchiveResult {
  archivePath: string;          // absolute
  slug: string;                 // final slug (after collision resolution)
  meta: ArchiveMeta;
}

export interface PruneFilter {
  beforeDate?: Date;           // ISO-parsed
  keepLast?: number;            // 0 = delete all
  status?: "completed" | "failed";
}

export interface PruneCandidate {
  slug: string;
  archivedAt: Date;
  meta: ArchiveMeta | null;     // null for legacy/partial archives
  reason: string;               // "older than <date>" | "beyond keep-last N" | "status filter"
}

export interface PruneResult {
  candidates: PruneCandidate[];
  deleted: string[];            // slugs actually deleted (empty in dry-run)
}

export interface ArchiveOptions {
  archivedBy: ArchivedBy;       // see types.ts addition
  force?: boolean;              // allow paused/escalated
  slug?: string;                // override auto-derived slug (still timestamped)
  now?: () => Date;             // test hook
}

export async function archiveCurrentRun(projectDir: string, opts: ArchiveOptions): Promise<ArchiveResult>;
export async function listArchives(projectDir: string): Promise<ArchiveMeta[]>;
export async function readArchive(projectDir: string, slug: string): Promise<ArchiveMeta>;
export async function pruneArchives(projectDir: string, filter: PruneFilter, opts: { dryRun: boolean }): Promise<PruneResult>;
export async function detectStaleTerminalState(projectDir: string): Promise<{ stale: boolean; status?: "completed" | "failed"; slug?: string; state?: OrchestratorState }>;
export async function finalizePartialArchive(projectDir: string): Promise<void>;
export function deriveSlug(state: Pick<OrchestratorState, "branch" | "feature" | "status">, override?: string, now?: () => Date): string;
```

**`deriveSlug` algorithm** (matches spec):
1. Let `base` = override (if provided & non-empty) — else strip `BRANCH_PREFIX` ("conduct/") from `state.branch`. If branch is empty/`"main"`, fallback to `slugify(state.feature, 40)`. If feature is also empty, fallback to `"run"`.
2. Slugify: lowercase, replace non-`[a-z0-9-]` with `-`, collapse runs of `-`, trim leading/trailing `-`, clamp to 40 chars.
3. If `state.status === "failed"`, append `-FAILED`.
4. Append `-YYYYMMDD-HHMMSS` using UTC from `now()` (default `Date`).
5. Collision suffix (`-2`, `-3`, …) is decided in `archiveCurrentRun` where we have filesystem access — `deriveSlug` is pure.

Max length guard: after (1)–(4), if total > 80 chars, truncate the slug-core portion until the whole string (including `-FAILED-…` suffix) fits in 80.

**`archiveCurrentRun`:**

1. Read state via `readStateLenient` (uses `validateStateJsonLenient`; tolerates missing/partial fields).
2. Guard: if `!opts.force && (state.status === "paused" || state.status === "escalated")`, throw `ArchiveRefusedError` with a message instructing the user to resume or pass `--force`.
3. Compute slug via `deriveSlug`, then resolve collision by appending `-2`, `-3`, … until the target path does not exist (bounded at 99 to avoid runaway; throw if exhausted).
4. `mkdirSecure(archivePath, { recursive: true })` — 0o700, umask-safe.
5. Write the `.archive-in-progress` marker atomically: `writeJsonAtomic(markerPath, "", { fsync: false })`.
6. **Build the work-list of entries to move (Codex round-4 explicit algorithm):**
   - Start with `FILES_TO_ARCHIVE` — each entry is a literal path relative to `.conductor/`.
   - THEN `fs.readdir(conductorDir)` and add any entry matching `ARCHIVE_PLAN_GLOB` (`/^plan-v\d+\.md$/`) to the work-list. This captures `plan-v1.md`, `plan-v10.md`, etc., including gaps. Non-matching `plan-notes.md` / `plan-draft.md.bak` stay at root.
   - Deduplicate (defense against a future FILES_TO_ARCHIVE entry that overlaps with the glob).
7. For each entry in the work-list:
   - If `lstat(src)` succeeds (use `lstat` so symlinks don't deref-then-fail), move it:
     - Try `fs.rename(src, dst)`.
     - On `EXDEV` (cross-filesystem), fall back to `fs.cp(src, dst, { recursive: true, force: false, errorOnExist: true, verbatimSymlinks: true })` followed by `fs.rm(src, { recursive: true, force: true })`. **Do NOT set `preserveTimestamps: true`** — some filesystems (certain FUSE/container mounts) reject the utimes call (Codex round-1 suggestion). Timestamps aren't part of AC #9; permissions are.
     - Post-move permissions: for the regular rename path, kernel preserves mode automatically. For the EXDEV fallback, explicitly normalize: walk the destination tree, `lstat` each entry, skip symlinks (chmod would follow the link and mutate the target), `chmod` regular files to 0o600, regular dirs to 0o700.
     - Ensure parent dir of `dst` exists first via `mkdirSecure(dirname(dst), { recursive: true })`.
8. For each entry in `FILES_TO_DELETE_ON_ARCHIVE` — `fs.rm(path, { force: true, recursive: false })`. (We do NOT touch lock files — see §1 item 11.)
9. Build `ArchiveMeta` from the loaded state; write via `writeJsonAtomic(getArchiveMetaPath(projectDir, slug), JSON.stringify(meta, null, 2) + "\n")`.
10. Delete the `.archive-in-progress` marker via `fs.rm(marker, { force: true })`. Archive is now durable and discoverable.
11. Return `ArchiveResult`.

Any failure between steps 4 and 10 leaves the marker in place → next `finalizePartialArchive` cleans up.

**`detectStaleTerminalState`:**
- If `state.json` missing → `{ stale: false }`.
- If read/parse fails → `{ stale: false }` (log the failure to stderr but do not throw — the user shouldn't be blocked from starting a new run because of a broken old state file; they'll see the stale files and we just won't auto-archive on this path).
- If `state.status ∈ {completed, failed}` → `{ stale: true, status, slug: deriveSlug(state), state }`.
- Else → `{ stale: false }`.

**`finalizePartialArchive`:**
- If `getArchiveDir(projectDir)` doesn't exist → return.
- **Safety check (Codex round-2):** `lstat(getArchiveDir(projectDir))` — if it's a symlink or not a directory, log to stderr and return WITHOUT writing. This prevents archival from following a user-created symlink outside `.conductor/` and writing archives to an unexpected location.
- For each direct child entry:
  - **Skip any entry whose name matches the partial-quarantine regex** `/-PARTIAL(?:-\d+)?$/` (Codex round-5 follow-up — covers both `-PARTIAL` and `-PARTIAL-2`/`-PARTIAL-3`/etc., since quarantine collisions can produce those suffixes and any of them can retain a stuck marker after manual tampering or failed marker-delete).
  - If it contains `.archive-in-progress`:
    - If `_archive-meta.json` also exists → delete the marker (archive was 99% done, just finalize).
    - Else → rename the dir to `<slug>-PARTIAL`. **Quarantine-collision handling (Codex round-2):** if `<slug>-PARTIAL` already exists, try `<slug>-PARTIAL-2`, `<slug>-PARTIAL-3`, ..., bounded at 99. If exhausted, log to stderr and leave the marker in place (user can inspect manually). Delete the marker AFTER the successful rename — the `-PARTIAL` suffix is now the quarantine signal. Belt-and-suspenders: both the ending-suffix skip-rule above AND marker deletion.
  - Log via stderr (no Logger dependency in archiver).

Extend the same archive-root symlink check to `archiveCurrentRun` before writing the marker (step 5) and to `listArchives` / `pruneArchives` before walking the dir.

**`listArchives`:**
- Read `getArchiveDir(projectDir)`; if missing, return `[]`.
- For each child:
  - If it's a file (not a dir), skip.
  - If it contains `.archive-in-progress`, skip (it's a partial).
  - If it contains `_archive-meta.json`, parse and return as `ArchiveMeta`. Use Zod-style defensive reads: required fields (`archive_version`, `archived_at`, `original_slug`, `final_status`) must be present; others get sensible defaults. On parse failure, synthesize a stub meta with `archive_version: 0`, `original_slug: <dir-name>`, `final_status: "unknown" as any`, `summary: null`.
  - If neither marker is present, synthesize a **legacy-stub** `ArchiveMeta` (`archive_version: 0`, `original_slug: <dir-name>`, `final_status: "unknown"`, `archived_at: <dir mtime>.toISOString()`, `summary: null`, `conductor_version: "legacy"`).
- Sort by `archived_at` descending and return.

**`readArchive`:**
- Uses same logic as `listArchives` for a single slug. Throws `ArchiveNotFoundError` if the directory is missing. For legacy archives, returns the stub (so `inspect` degrades gracefully).
- Validates `slug` via `validateIdentifier` before touching the filesystem — rejects path traversal at the API boundary.

**`pruneArchives`:**
- Error if neither `beforeDate` nor `keepLast` is set (caller's CLI layer should already enforce; this is belt-and-suspenders).
- List archives (sorted desc by date).
- If `keepLast !== undefined`: mark archives with index ≥ keepLast for deletion (so `keepLast: 0` deletes all).
- If `beforeDate`: also include archives older than that.
- If `status`: keep-set is further narrowed to only archives matching that status (legacy archives with `final_status: "unknown"` are treated as not matching the `status` filter, so they are NOT deleted if the user passed `--status`. This is safer — the user explicitly asked for a status, and we don't know what a legacy archive was).
- In dry-run: return candidates with no deletion.
- Else: `fs.rm(archivePath, { recursive: true, force: true })` for each candidate; collect successful slugs into `result.deleted`.

### 2.2 `src/core/archiver.test.ts`

Unit tests using `os.tmpdir()` sandboxes:

- **`deriveSlug`**
  - branch present → slug stripped of prefix
  - branch missing → slug from feature
  - branch `"main"` → slug from feature

- **`archiveCurrentRun`** (happy path)
  - Seed `.conductor/` with every entry in `FILES_TO_ARCHIVE` (some dirs, some files, some both present and absent), a fake state.json, and a couple of `FILES_TO_KEEP_AT_ROOT` entries.
  - After: archive dir contains all present entries with matching names/types; root still has the keep-list entries; marker is gone; `_archive-meta.json` exists and parses.
- `archiveCurrentRun` with `state.status = "paused"` and `force: false` → throws
- `archiveCurrentRun` with `state.status = "paused"` and `force: true` → succeeds
- `archiveCurrentRun` with missing `state.json` → throws with a clear message
- `archiveCurrentRun` collision: pre-create a dir matching the derived slug; call with deterministic `now`; expect `-2` suffix
- `archiveCurrentRun` preserves permissions: seed file with 0o600, seed dir with 0o700, verify after archival the destination keeps them (AC #9)
- **`detectStaleTerminalState`**
  - No state.json → `{ stale: false }`
  - `status: "completed"` → stale=true, status=completed, slug set
  - `status: "failed"` → stale=true, status=failed
  - `status: "paused"` → `{ stale: false }` (paused is resumable, not stale)
  - `status: "executing"` → `{ stale: false }`
  - Corrupt JSON → `{ stale: false }` (does not throw)
- **`finalizePartialArchive`**
  - No archive dir → no-op
  - Archive with marker + meta → marker removed, dir kept under original slug
  - Archive with marker, no meta → dir renamed to `<slug>-PARTIAL`, **marker deleted AFTER the rename** (the `-PARTIAL` suffix is the quarantine signal; matches §2.1 implementation contract, Codex round-4 alignment)
  - Archive without marker → untouched (so legacy archives aren't disturbed)
  - Archive dir named `foo-PARTIAL` with a stuck marker → skipped (suffix check prevents `foo-PARTIAL-PARTIAL`)
  - Quarantine collision: seed `<slug>-PARTIAL` AND a partial archive with marker → rename to `<slug>-PARTIAL-2`
  - Archive-root symlink → stderr log, function returns without traversing
  - Multiple `plan-v*.md` files with gaps (v1, v3, v10) → the regex `/^plan-v\d+\.md$/` matches all three; non-matching `plan-notes.md` stays at root
- **`listArchives`**
  - Empty / missing dir → `[]`
  - Mix of valid, legacy, and partial entries → returns valid + legacy (synthesized stub), skips partials
  - Sorted descending by `archived_at`
- **`readArchive`**
  - Valid slug → returns parsed meta
  - Legacy slug (no meta) → returns stub
  - Missing slug → throws `ArchiveNotFoundError`
  - Invalid identifier (contains `/` or `..`) → throws before touching fs
- **`pruneArchives`**
  - `--keep-last 0 --dry-run` with 3 archives → 3 candidates, 0 deleted
  - `--keep-last 2 --yes` with 3 archives → oldest 1 deleted
  - `--before <date>` → only archives older than date are candidates
  - `--status failed` → legacy archives (unknown status) are NOT selected
  - Dry-run never mutates filesystem (verify archive dirs still exist after)

### 3.4 `src/cli.ts`

**Version bump:** `.version("0.7.5")` → `.version("0.7.6")`.

**Start command** — two hooks added after `acquireProcessLock` succeeds, before `new Orchestrator(options)`:

**Hook 1: Post-lock resumable-state + corrupt-state guard (spec Trigger Matrix rows 3-4).** `conduct start` today does NOT check for existing `state.json` — it just overwrites via `StateManager.initialize()`. Per spec: if a `paused` or `escalated` state is present, refuse and direct the user to resume or force-archive. Additionally (Codex round-3 finding): if `state.json` is corrupt/invalid, refuse with recovery instructions — silent continue would overwrite the broken run and mix old run artifacts with new ones.

**Prerequisite: tighten `readState()` (Codex round-5 finding).** Today at `src/cli.ts:61-68`, ALL `fs.readFile` failures (ENOENT, EACCES, EIO, etc.) collapse into `{ status: "missing" }`. That's dangerous for the new guards: a permission error on `state.json` would be misclassified as "no state", letting Hook 1 proceed into `StateManager.initialize()` and clobber a protected run. Change `readState()` to:

```ts
type ReadStateResult =
  | { status: "ok"; state: OrchestratorState }
  | { status: "missing" }
  | { status: "invalid"; errors: string[] }
  | { status: "unreadable"; error: string };

async function readState(projectDir: string): Promise<ReadStateResult> {
  const statePath = getStatePath(projectDir);
  let raw: string;
  try {
    raw = await fs.readFile(statePath, "utf-8");
  } catch (err) {
    const code = (err as NodeJS.ErrnoException)?.code;
    if (code === "ENOENT") return { status: "missing" };
    return { status: "unreadable", error: `${code ?? "?"}: ${String(err)}` };
  }
  const result = validateStateJsonLenient(raw);
  if (!result.valid) return { status: "invalid", errors: result.errors };
  return { status: "ok", state: result.state as OrchestratorState };
}
```

Hook 1 handles all four read results. **Only `missing`, `ok+completed`, and `ok+failed` are allowed to proceed into `StateManager.initialize()`** (Codex round-6 critical finding). Every other classification aborts start with clear recovery guidance:
- `missing` → proceed (nothing to guard).
- `ok` + `completed` / `failed` → proceed; Hook 2 will auto-archive.
- `ok` + `paused` / `escalated` → exit 2 with "resume or archive --force" guidance.
- `ok` + `initializing` / `questioning` / `planning` / `executing` / `reviewing` / `flow_tracing` / `checkpointing` → exit 2 with "this is a stale crashed run; resume --force-resume or archive --force --yes" guidance. Do NOT fall through — the previous draft let these statuses through and Hook 2 would ignore them (only archives terminal), leaving `StateManager.initialize()` to clobber the artifacts.
- `invalid` → exit 1 with Zod errors + manual-quarantine guidance.
- `unreadable` → exit 1 with "cannot read state.json; check filesystem permissions / disk / locks" guidance.

All four already-existing callsites (`status`, `resume`, `pause` commands + the manual `archive` command) need to handle `unreadable` too — plan: treat it identically to `invalid` in those callsites (hard exit with clear message). Update `src/cli.test.ts` expectations accordingly.

**Exit-code contract (Codex round-6 drift fix):** the spec §CLI Surface for `conduct archive` uses exit code 2 for "run is resumable and `--force` not set". We extend that to all non-terminal states in Hook 1 for consistency (exit 2 = "run exists and needs user decision to unblock"). Exit 1 is reserved for "something is broken" (invalid/unreadable state). So:
- Non-terminal but valid-read status (paused/escalated/executing/etc.) → **exit 2**.
- Invalid or unreadable state.json → **exit 1**.

```ts
// Post-lock guard. Split by state class so the message templates don't
// drift and each recovery path is verified for the case it covers.
const preStartState = await readState(projectDir);

async function exitReleasingLock(code: number): Promise<never> {
  if (releaseLock) { try { await releaseLock(); } catch { /* best effort */ } }
  process.exit(code);
}

if (preStartState.status === "unreadable") {
  // Codex round-5: this branch exists because readState() now distinguishes
  // ENOENT from EACCES/EIO/etc. DO NOT fall through into StateManager.initialize().
  console.error(chalk.red(
    `\nCannot read .conductor/state.json: ${preStartState.error}\n\n` +
    `This is not a missing-file case — check filesystem permissions,\n` +
    `disk health, or whether another process has the file open.\n`
  ));
  await exitReleasingLock(1);
}

  // Recovery here is MANUAL quarantine — `conduct archive --force` won't
  // work because archiveCurrentRun() re-reads state via the same lenient
  // validator and would throw on the same corrupt input.
  console.error(chalk.red(
    `\nExisting .conductor/state.json is invalid and cannot be classified:\n` +
    preStartState.errors.map((e) => `  - ${e}`).join("\n") + "\n\n" +
    `Options:\n` +
    `  - Inspect manually: ls .conductor/ && cat .conductor/state.json\n` +
    `  - Quarantine:       mv .conductor .conductor-corrupt-$(date +%s)\n` +
    `  - Start fresh (destructive): rm -rf .conductor/ (loses prior run)\n`
  ));
  await exitReleasingLock(1);
}

if (preStartState.status === "ok") {
  const s = preStartState.state.status;
  if (s === "paused" || s === "escalated") {
    // Recovery: conduct resume OR conduct archive --force (state is
    // valid, just not terminal). --force overrides the paused/escalated
    // refusal in archiveCurrentRun.
    console.error(chalk.red(
      `\nA prior '${s}' conductor run is present in this project.\n` +

    ));

  }
  // Codex round-6: block ALL other non-terminal states too. Falling
  // through would let Hook 2 ignore them (it only auto-archives
  // completed/failed), and then StateManager.initialize() would

  const NON_TERMINAL_STUCK: OrchestratorStatus[] = [

      `  - Archive + start:  conduct archive --force --yes --project "${projectDir}" && conduct start ...\n`
    ));

  }
  // Only `completed` and `failed` reach here — Hook 2 will auto-archive.
}
// status === "missing" falls through: nothing to guard.
```

### 3.5 `src/core/orchestrator.ts`

**Critical ordering constraint (Codex round-1 finding):** `eventLog.stop()` flushes buffered events and will re-create `.conductor/` + re-append to `events.jsonl` if called AFTER archival (see `EventLog.writeEvents()` at `src/core/event-log.ts:199-228` — it calls `mkdirSecure(conductorDir)` and `fs.appendFile(this.logPath, ...)`). The current `run()` finally block at line 689-692 calls `this.eventLog.stop()` AFTER the try/catch completes. If archival runs inside `complete()` or inside the `catch` block, the archive will be "complete" and then the finally-block flush will leave a stray root `events.jsonl` behind. Same reasoning applies to `Logger` with its open write stream on `logs/conductor.log`.

**Resolution: move archival out of `complete()` and the `catch` block, and into a new terminal-archive step that runs AFTER `eventLog.stop()` + `logger.close()`.**

Instead of archiving inside `complete()` / `catch`, do this:

1. Add a private field `this.terminalArchivalReason: ArchivedBy | null = null`.
2. In `complete()`: at the end, set `this.terminalArchivalReason = "auto-on-completion"`. No archival call.
3. In the `catch(err)` at line 675-688: after `setStatus("failed")`, set `this.terminalArchivalReason = "auto-on-failure"`. Still `throw err` as before.
4. Change the `finally` block to:

```ts
} finally {
  // Flush and close ALL write paths before we potentially move files.
  // Order: eventLog (re-creates .conductor/ on flush), then logger
  // (holds a WriteStream on logs/conductor.log).
  try { await this.eventLog.stop(); } catch { /* best effort */ }
  try { await this.logger.close(); } catch { /* best effort */ }

  // v0.7.6: Auto-archive AFTER all writers are closed so archival is the
  // final mutation on .conductor/. Archival failure must NEVER override
  // the in-flight error from the try block (hence wrapped in its own try).
  if (this.terminalArchivalReason) {
    try {
      const result = await archiveCurrentRun(this.options.project, {
        archivedBy: this.terminalArchivalReason,
      });
      // Logger is closed; write to stdout directly. Use a plain
      // console.log so the user gets a breadcrumb to the archive path.
      console.log(chalk.gray(`\n  Run archived to .conductor/archive/${result.slug}/\n`));
    } catch (archiveErr) {
      console.warn(chalk.yellow(
        `Auto-archival failed (terminal state already recorded): ${archiveErr instanceof Error ? archiveErr.message : String(archiveErr)}`
      ));
    }
  }
}
```

**`archiveCurrentRun` is imported at the top of `orchestrator.ts`** — not via `await import()`. Dynamic import adds loader-cost variance and makes mocking harder; since the module is always needed in terminal paths, hoist it.

**`Logger.close()` must become awaitable** (Codex round-2 finding). Today at `src/utils/logger.ts:88-97` it calls `logStream.end()` and returns `void` — the stream is still draining when the method returns. If archival hits the EXDEV fallback (`fs.cp` while bytes are still in the writable's buffer), tail bytes can be lost from the archived `conductor.log`. Change the signature:

```ts
// src/utils/logger.ts
close(): Promise<void> {
  if (this.closed) return Promise.resolve();
  this.closed = true;
  if (this.exitHandler) {
    process.removeListener("exit", this.exitHandler);
    this.exitHandler = null;
  }
  return new Promise<void>((resolve) => {
    // fs.WriteStream.end(cb) invokes cb after the stream is fully flushed
    // and closed. No reject path — stream close errors are not actionable
    // at this point (file is already on disk or isn't).
    this.logStream.end(() => resolve());
  });
}
```

All existing non-awaiting callers (`shutdown()` at line 2364, the process-exit handler at logger.ts:80-81, and `logger.test.ts`) remain functionally correct — they just don't await. The process-exit handler path is a safety net and the event loop exits soon after; best-effort flush is acceptable there. `shutdown()` should ALSO be updated to `await this.logger.close()` so the graceful-shutdown path has a real flush barrier — zero functional cost.

Why this works:
- `eventLog.stop()` flushes before archival, so no buffered events can sneak in afterward.
- `logger.close()` ends the write stream before `logs/` is moved. Subsequent writes from archival go through stdout directly.
- Archival is the very last step in `run()` — nothing after it can recreate `.conductor/` files.
- `ConductorExitError` also propagates through `finally`; the trigger-reason is only set on `completed`/`failed`, so `ConductorExitError` paths (which go through the catch re-throw without setting `terminalArchivalReason`) do NOT archive. Correct per spec AC #10 (escalated → no archival).

**Additional `complete()` vs `git commit` ordering:** `complete()` calls `git commit` at line ~2386. The commit is part of the finished run and should land BEFORE archival. Since archival moves to `finally`, the git commit naturally precedes it. Good.

**Logger idempotency (confirmed):** `Logger.close()` at `src/utils/logger.ts:89-97` checks `if (this.closed) return;` first, so double-close is a no-op. Safe.

**`shutdown()` interaction:** `shutdown()` at line 2356-2367 also calls `this.eventLog.stop()` and `this.logger.close()`. Both are already idempotent (EventLog.stop() returns early if not started; Logger.close() returns early if closed). `shutdown()` is only invoked via SIGINT/SIGTERM at the CLI layer, which bypasses normal run() cleanup — archival does NOT run from shutdown path, so SIGINT'd runs stay in the root `.conductor/` until the next `conduct start` auto-archives them. Spec-compliant (shutdown is interrupted, not terminal).

## 6b. Spec-vs-plan lock-file reconciliation (Codex round-1)

The authoritative spec at `.claude/specs/v0.7.6-auto-archival.md` line 130 currently lists `conductor.lock` and `conductor.lock.info` in the "Files that are DELETED" block. That is operationally wrong — during start-time archival the CLI is actively holding the lock, and the orchestrator's post-run archival runs under the CLI's lock too. Deleting lock files mid-operation would race `releaseLock()` and could leave the `.lock/` directory (proper-lockfile's state) in a weird state.

**Decision: update the spec to match the plan.** In step 12 of the implementation order, edit `.claude/specs/v0.7.6-auto-archival.md` lines 128-133 to:

```
### Files that are DELETED (not archived)

pause.signal            (if present — transient)
# (lock files are left in place; releaseLock() handles them)
```

This keeps the spec authoritative and prevents a future change from re-introducing the deletion based on the old spec text.
