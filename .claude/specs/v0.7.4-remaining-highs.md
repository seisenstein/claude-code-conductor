## Fix 1 — A-1: Migrate 9 non-atomic writer sites to `writeJsonAtomic`

**Finding**: `writeJsonAtomic` was introduced in v0.7.3 as part of H-14 but applied only to the 2 escalation sites. The v0.7.3 audit's non-atomic writer inventory found 9 additional sites where a crash mid-write corrupts orchestration state and the data is not reconstructible.

### Sites to migrate

| # | File:line | Current call | Risk |
|---|-----------|--------------|------|
| 1 | `src/mcp/tools.ts:502` | `fs.writeFile` in `handleClaimTask` | task file partial write on crash during claim |
| 2 | `src/mcp/tools.ts:688` | `fs.writeFile` in `handleCompleteTask` | task file partial write on completion |
| 3 | `src/mcp/tools.ts:808` | `fs.writeFile(filePath, "{}", ...)` contract stub | two-step write race with line 811 |
| 4 | `src/mcp/tools.ts:811` | `fs.writeFile` contract full write | partial contract visible between two writes |
| 5 | `src/core/state-manager.ts:383` | `fs.writeFile` in `createTask` | task file partial write on create |
| 6 | `src/core/state-manager.ts:413` | `fs.writeFile` dependency block write | dep state corruption on crash |
| 7 | `src/core/state-manager.ts:650` | `fs.writeFile` in `resetOrphaned` | orphan-reset partial write |
| 8 | `src/utils/known-issues.ts:49` | `writeFileSecure` in `saveKnownIssues` | security findings registry corruption |
| 9 | `src/core/event-log.ts:226` | `fs.writeFile` in rotation overwrite | entire event log lost on crash during rotation |

### Approach

Replace each site with `writeJsonAtomic(destPath, content)`. The helper already handles:
- tmp+rename atomicity
- fsync by default (durability)
- chmod enforcement to `SECURE_FILE_MODE` (0o600)

Two sites have nuance:

**Sites #3 + #4 (contract stub-then-full)**: merge into a single `writeJsonAtomic` write. The stub-then-full pattern was for lock initialization; with atomic write, it's unnecessary.

**Site #9 (event-log rotation)**: inside the rotation code path, replace the two `fs.writeFile` calls (one for new content, one for empty-on-failure fallback at line 229) with a single `writeJsonAtomic`. Remove the empty-on-failure fallback — leaving the existing file intact is strictly better than truncating on rotation failure.

### Backward compatibility

`writeJsonAtomic` uses `fs.rename` which fails across filesystems; the inputs are all within `.conductor/` (same FS) so this is not a risk. All callers must await the returned promise (they already do for `fs.writeFile`).

### Tests (T-5 — 4 tests in `secure-fs.test.ts`)

Since `writeJsonAtomic` itself ships in v0.7.3 without behavioral tests for the helper, add them here:

1. **tmp+rename path** — write content, assert dest contains it, assert tmp gone.
2. **fsync called by default** — spy on FileHandle.sync, assert it fires.
3. **fsync:false skips sync** — spy, assert not called.
4. **mode defaults to 0o600** — stat the result, assert mode bits.

Site-level regression tests are out of scope (swaps are 1:1 mechanical).

### Commit

`fix(persistence): migrate 9 non-atomic writer sites to writeJsonAtomic [A-1]`

Plus: `test(secure-fs): cover writeJsonAtomic contract [T-5]` (can be in the same commit since the helper tests are directly related).

## Fix 3 — A-3: Event-log applies `redactSecrets()` to worker failure messages

**Finding**: `src/core/event-log.ts:205` appends events via `fs.appendFile` without passing event `error` strings through `redactSecrets()` (defined in `logger.ts:20-37` as part of v0.7.2 H-1). Worker failure events (`worker_fail`, `task_failed`, `worker_timeout`) commonly carry stderr/stack-trace text that can contain API keys, bearer tokens, or other credentials — which would then persist in `.conductor/events.jsonl` plaintext on disk.

### Approach

Before JSONL serialization at `event-log.ts:205`, map each event's `error` field through `redactSecrets()`. Only apply to events that have an error field; leave non-error events untouched to avoid paying the regex cost on every event.

```ts
// event-log.ts — top of writeEvents or equivalent:
import { redactSecrets } from "../utils/logger.js";

const sanitized = events.map((e) => {
  if (!("error" in e) || typeof e.error !== "string") return e;
  return { ...e, error: redactSecrets(e.error) };
});
const lines = sanitized.map((e) => JSON.stringify(e)).join("\n") + "\n";
await fs.appendFile(this.logPath, lines, { encoding: "utf-8", mode: 0o600 });
```

Same treatment for the rotation path at line 226 — read the existing content, re-serialize through the sanitizer, then write via `writeJsonAtomic`.

### Tests

Add 2 cases to `src/core/event-log.test.ts`:
1. **Event with Bearer token in error gets redacted** — append an event with `error: "Bearer sk-ant-xxxxxxxxxxxxxxxx fetch failed"`; read back; assert the token is `***REDACTED***`.
2. **Benign events unchanged** — append an event without `error`; assert the payload is identical byte-for-byte.

### Commit

`fix(security): redact secrets in event log error fields [A-3]`

## Fix 4 — A-4: MCP `task_id` rejects forward slashes (path traversal)

**Finding**: `src/utils/validation.ts:25-26` documents that "the calling code in tools.ts additionally rejects forward slashes to ensure task IDs are simple identifiers, not paths." That check does not actually exist. `src/mcp/tools.ts:441` only calls `validateFileName(input.task_id)`, which permits forward slashes by design (for `files_changed` paths). A task_id like `"subdir/task-001"` passes validation; subsequent `path.join(dir, ${task_id}.json)` writes under a subdirectory.

### Approach (scope finalized after Codex spec-review round 2)

Round 1 narrowed scope to task_id only; round 2 revealed additional filesystem-bound identifiers that also need stricter validation. Final scope:

**In scope for `validateIdentifier`** (all become filesystem path segments):
- `task_id` — becomes `${task_id}.json` (`tools.ts:449`, `tools.ts` in handleCompleteTask)
- dependency `task_id` inside claim_task loop (`tools.ts:481-489`)
- `session_id` — becomes a directory name at `path.join(dir, input.session_id, SESSION_STATUS_FILE)` (`tools.ts:749`)
- `contract_id` — becomes `${contract_id}.json` (`tools.ts:799`). Round-2 finding: the docs claim endpoint-style IDs like `'POST /api/users'` are supported (`coordination-server.ts:300`), but the storage code uses the raw ID as a filename. A raw `/` would try to write to a non-existent subdirectory and fail with ENOENT. The aspirational doc is actively broken — this is a latent bug, not a supported feature. The right fix is to reject `/` in `contract_id` and update the doc text to match.

**Explicitly NOT changed**:
- `files_changed` validation — paths, not identifiers.

Add `validateIdentifier(id)` in `validation.ts`:

```ts
// validation.ts — add near validateFileName:
/**
 * Validates an identifier that becomes part of a filesystem path (task_id,
 * session_id, contract_id, dependency task_id). Stricter than validateFileName:
 * rejects all path separators to ensure the resulting path does not traverse.
 *
 * NOT for: `files_changed` (intentional relative paths); any free-form
 * content field.
 */
export function validateIdentifier(id: string): FileNameValidationResult {
  const base = validateFileName(id);
  if (!base.valid) return base;
  if (id.includes("/") || id.includes("\\") || id.includes(":")) {
    return {
      valid: false,
      reason: "Identifier cannot contain path separators (/, \\, :)",
    };
  }
  return { valid: true };
}
```

### Migration checklist

| Handler | Field | Current | Change |
|---------|-------|---------|--------|
| `handleClaimTask` (`tools.ts:441`) | `task_id` | `validateFileName` | `validateIdentifier` |
| `handleClaimTask` (`tools.ts:481-489`) | dep `task_id` | `validateFileName` | `validateIdentifier` |
| `handleCompleteTask` | `task_id` | `validateFileName` | `validateIdentifier` (grep for exact line) |
| `handleGetSessionStatus` (`tools.ts:743`) | `session_id` | `validateFileName` | `validateIdentifier` |
| `handleRegisterContract` (validates via handler) | `contract_id` | `validateFileName` | `validateIdentifier` |

### Documentation update

`src/mcp/coordination-server.ts:300` — update the tool schema description:

```ts

// Becomes:
```

## Fix 7 — A-7: Synthetic flow-finding suppression + escalation (redesigned round 1)

**Finding**: H-4 (v0.7.3) synthesizes a `FlowFinding` when `flowReview()` rejects, forcing another cycle. Two problems:
1. `createFixTasksFromFindings` at `orchestrator.ts:2824` creates a new task every cycle even though no fix task can resolve an infrastructure failure (git unavailable, tracer crashes).
2. Looping forever on infrastructure failures wastes API budget and pollutes task lists.

### Design (revised after Codex spec-review round 1)

Round-1 review pointed out three defects in the original design:
- `flow_id` is **not** persisted as a structured task field — only interpolated into the task description. So "dedup by flow_id" is not implementable without a task schema change.
- The synthetic `flow_id` is cycle-scoped (`flow-tracing-failure-cycle-N`), so dedup "same flow_id" would never fire across cycles anyway.
- Resetting the counter on fulfilled-null is risky — null includes intentional skips (`--skip-flow-review`, empty diff, capacity defer), which would mask consecutive infra failures.

The **revised** design:

1. **Synthetic infrastructure findings do NOT generate fix tasks**. Add a centralized `isSyntheticFlowInfraFinding(f: FlowFinding): boolean` helper (in `orchestrator.ts` or `utils/types.ts`) that detects the synthetic-finding shape via **multiple markers**, not just a single string prefix (per Codex round-2 suggestion — a single-marker check is brittle):

   ```ts
   function isSyntheticFlowInfraFinding(f: FlowFinding): boolean {
     return (
       f.actor === "conductor" &&
       f.file_path === "<flow-tracing-infrastructure>" &&
       f.flow_id.startsWith("flow-tracing-failure-")
     );
   }
   ```

   Three-marker check is robust against future changes that alter one marker (e.g., localization of titles). Both `createFixTasksFromFindings` (to skip task creation) and the H-4 handler itself (for completeness) should use the same helper.
2. **Counter + escalation**: track `consecutive_flow_tracing_failures` on `OrchestratorState`. Increment inside the `settled[1].status === "rejected"` branch in `orchestrator.ts:474`. If the counter reaches 2, call `escalateToUser()` with a specific "flow-tracing infrastructure failed N times in a row" reason. Reset the counter when `flowReview()` returns a **non-null** `FlowTracingReport` (i.e., actual trace success — not skip). Skip paths (null) neither increment nor decrement.

Rationale: the synthetic finding still gets surfaced in logs and in the cycle record for observability — it just doesn't create a misleading fix task, and it doesn't loop forever.

### State-schema migration

Add `consecutive_flow_tracing_failures` field to `OrchestratorState`:

- `src/utils/types.ts` — `OrchestratorState`: add `consecutive_flow_tracing_failures: number`.
- `src/utils/state-schema.ts` — `OrchestratorStateSchema`: add `consecutive_flow_tracing_failures: z.number().int().nonnegative().default(0)`. Use the same `.default(0)` pattern as v0.7.2 `output_too_large_failures` and v0.7.3 `execution_errors`. Do NOT mark as `.optional()` — the default handles the backfill.
- `src/core/state-manager.ts` — `initialize()`: set to 0 on fresh run.

### Tests

Add 3 tests to `src/core/orchestrator.test.ts`:
1. **Synthetic infra finding does not create a fix task** — mock `flowReview` to reject; call `checkpoint`; assert no task with `task_type === "security"` titled "Fix: Flow tracing failed" was created. Assert the warning log fires.
2. **Counter increments on rejection, resets on real success, untouched on null** — simulate reject → reject (counter=2 triggers escalation, separate test) → fulfilled-report (counter resets to 0) → fulfilled-null (counter unchanged). This single test exercises the three branches.
3. **Counter at 2 triggers escalation** — simulate two consecutive rejects; assert `escalateToUser` called on the second with a reason containing "flow-tracing infrastructure".

Plus one state-schema test in `state-schema.test.ts`:
- **Backfill `consecutive_flow_tracing_failures` to 0** — pre-v0.7.4 state.json without the field loads and backfills. Same shape as existing `output_too_large_failures` backfill test.

`fix(orchestrator): suppress synthetic flow-infra fix tasks + escalate on repeated failure [A-7]`

## Fix 8 — A-8: Document `read_updates` mtime TOCTOU residual (no code change)

**Finding**: Report 03 M-4 notes that `read_updates`' mtime pre-filter has a TOCTOU race: a file's mtime is checked, then its content is read; a concurrent `appendFile` between those two steps means the updated content is present but the mtime-based skip decision was made on stale info. Practical impact: message may be returned on the *next* `read_updates` call instead of this one. Transient delivery delay, not data loss.

### Approach

No code change — fix would require restructuring the loop to stat-then-read as a coroutine or adopting file-level locking, both heavier than the bug warrants. Instead:

1. Add a comment block at `src/mcp/tools.ts` around the mtime pre-filter describing the TOCTOU, the observable effect, and why it's an acceptable residual.
2. Add a test that documents the expected behavior (no stronger guarantee): messages appearing after the mtime check but before the read *may* be returned in the current call or a subsequent call, but never lost.

### Tests

Add 1 test to `tools.test.ts`:
- **Concurrent appendFile during read_updates mtime check doesn't lose messages** — use a controlled timing harness (maybe via fake timers + promise ordering): call `handleReadUpdates({since: T})` and concurrently append; assert the message shows up in either the current call's return or a subsequent `handleReadUpdates` call. Document the TOCTOU as the reason we cannot assert which one.

### Commit

`docs(mcp): document read_updates mtime TOCTOU residual [A-8]`

## Decisions resolved after Codex spec-review round 2

1. **A-4 scope expanded** to include `session_id` (filesystem directory segment at `tools.ts:749`) and `contract_id` (filesystem filename stem at `tools.ts:799`). Round-1 deferred both; round-2 revealed both are vulnerable to the same path-traversal pattern.
2. **`contract_id` slash-support reversed**: the tool-schema description claimed endpoint-style IDs (`'POST /api/users'`) were supported, but storage code never handled the resulting `/` (would fail with ENOENT). The aspirational example was actively broken. Fix rejects `/` and updates the doc text to match reality.
3. **A-7 synthetic detection uses three-marker check**: single-prefix `flow_id.startsWith("flow-tracing-failure-")` is brittle. Centralized helper `isSyntheticFlowInfraFinding` checks `actor === "conductor"`, `file_path === "<flow-tracing-infrastructure>"`, and `flow_id` prefix together. Keeps all three shape invariants guaranteed at every call site.
