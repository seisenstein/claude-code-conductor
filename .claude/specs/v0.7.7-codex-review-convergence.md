# Spec: Port ClaudeCodexDiscussion convergence patterns into the conductor's Codex reviewer

**Target version:** v0.7.7
**Status:** Draft — pending Codex plan review
**Motivation:** The conductor's `CodexReviewer` uses thin, generic one-liner prompts (see `src/core/codex-reviewer.ts:396–483`). Real-world use produced drip-feeding (Codex holding findings for "next round"), weak adversarial depth, and frequent failure to converge within the 5-round discussion budget. The stand-alone `/Users/cameron/Documents/ClaudeCodexDiscussion` project has a refined prompt architecture whose slash commands (`/codex-review-plan`, `/codex-review-code`) consistently converge. This port brings the same prompt discipline to the conductor's non-interactive `codex exec` pipeline.

---

## Problem statement

Today, every review prompt the conductor sends is ~4 sentences + `JSON_FORMAT_INSTRUCTIONS`. Example, verbatim from `reviewPlan`:

```
Review this implementation plan for completeness, correctness, and potential issues.
For each issue, provide a clear description.
You have access to the `coordinator` MCP server with tools: get_tasks, get_contracts, get_decisions, read_updates.
Use these to understand the full context of the project's coordination state.
```

Consequences observed in prior conductor runs:

1. **Drip-feeding** — Codex often says "I'll look at X next round" on round 1 and burns the budget.
2. **Weak signal** — reviews mix nits with critical findings at the same severity.
3. **Non-adversarial baseline** — Codex defaults to "this looks reasonable" unless pushed to try to break it.
4. **No round awareness** — Codex doesn't know whether it's on the first round or the last, so it can't adapt how to spend its message.
5. **Vague verdicts** — Codex maps correctness failures to "minor" severity too often, because the category is not defined.

Meanwhile, `/Users/cameron/Documents/ClaudeCodexDiscussion` encodes proven mitigations in:
- `.claude/commands/codex-review-plan.md`
- `.claude/commands/codex-review-code.md`
- `src/review-runner.mjs` (`buildRoundBudgetBlock`, `buildReviewPrompt`)

## Goal

Adopt these mitigations inside `src/core/codex-reviewer.ts` **without changing the JSON-verdict contract** that `orchestrator.ts` parses downstream.

## Non-goals

- Do **not** change the Codex invocation mechanics (`codex exec --full-auto`, stdin payload, MCP wiring, model selection, timeouts, byte caps, rate-limit detection, retries).
- Do **not** introduce an MCP-based chat session (the conductor uses one-shot CLI invocations on purpose for stability + auditability).
- Do **not** change the orchestrator's round-loop bounds (`MAX_PLAN_DISCUSSION_ROUNDS = 5`, `MAX_CODE_REVIEW_ROUNDS = 5`).
- Do **not** touch the Claude-side "investigator" prompts (`orchestrator.ts:1279` and `:1932`). Those are separate work.

## Design

### New module: `src/core/codex-review-prompts.ts`

Pure-function prompt builders. No I/O, no side effects. Exports:

```ts
export interface RoundBudget {
  current: number;   // 1-indexed; initial review is round 1
  softCap: number;   // preferred total Codex invocations
  hardCap: number;   // absolute upper bound; may equal softCap
}

export function buildAdversarialStance(target: "plan" | "code"): string;
export function buildFeedbackFraming(): string;
export function buildRoundBudget(budget: RoundBudget): string;
//   Emits three banner modes:
//     current < softCap       → normal budget block
//     current === softCap     → "FINAL PLANNED ROUND" banner
//     current > softCap       → "OVERTIME" banner (defensive; dead in practice
//                                because the orchestrator stops invoking Codex
//                                once discussionRound === MAX_*_ROUNDS, but the
//                                builder supports it for safety).
export function buildSeverityTaxonomy(target: "plan" | "code"): string;
```

Content is adopted from ClaudeCodexDiscussion (see `.claude/commands/codex-review-plan.md` lines 70–128 and `src/review-runner.mjs` lines 52–81, 151–169), adapted for the conductor's JSON-verdict output format.

### Severity taxonomy mapping

ClaudeCodexDiscussion uses rich free-text labels. The conductor's JSON schema uses `severity: "minor" | "major" | "critical"`.

Reconcile by asking Codex to:
- Prefix every `description` string with one category tag (e.g. `"[CRITICAL] Missing auth check at handlers/posts.ts:42 — ..."`)
- Set `severity` to the mapped bucket per the taxonomy:

| Category (plan + code) | `severity` bucket |
|---|---|
| CRITICAL | `critical` |
| CORRECTNESS | `major` |
| ARCHITECTURE (code only) | `major` |
| SECURITY (code only) | `major` (or `critical` if exploitable) |
| ROBUSTNESS (code only) | `major` |
| COMPLETENESS (plan only) | `major` |
| ORDERING (plan only) | `major` |
| RISK (plan only) | `major` |
| ALTERNATIVE (plan only) | `minor` |
| SUGGESTION | `minor` |
| QUESTION | `minor` |
| NIT | `minor` |
| PRAISE | **not put in `issues` array — goes in `summary` instead** |

The parser is already tolerant (descriptions are free text), so non-compliant Codex responses degrade gracefully to the current behavior.

### Fix escalation counting so non-actionable categories can't trigger false escalations

**[CRITICAL finding from plan review:]** Today `orchestrator.ts:1243` and `orchestrator.ts:1902` feed every `reviewResult.issues` entry into the `issueCounts` map; after `MAX_DISAGREEMENT_ROUNDS` (2) recurrences of the same 80-char key, the conductor escalates to the user. With the new taxonomy, a persistent [QUESTION] or [NIT] could look identical to persistent disagreement on a critical bug.

Fix in both loops (belt-and-suspenders with the prompt rule above):

- Only increment `issueCounts` for issues whose string starts with `[critical] ` or `[major] ` (the prefix `codex-reviewer.ts` already prepends via `[${severity}] ...` at line 883). `minor` issues are still surfaced in the logs, just not eligible to trip escalation.
- Add a comment pointing at the taxonomy as the source of the convention.

This is a small change (~6 lines in orchestrator.ts) but critical for soundness.

### Enforce verdict ↔ issues consistency in the parser

**[Round-4 finding from Codex (CRITICAL):]** Nothing in the current design catches a contradictory response: `verdict: "APPROVE"` alongside `issues: [{ severity: "critical", ... }]`. With the new short-circuit layering, this response would exit the loop and propagate approval downstream — even though Codex itself flagged blockers.

Fix inside `parseStructuredResponse` (post-parse, pre-return):

- After building the `issues` string array, check `verdict === "APPROVE" && hasBlockingIssues(issues)`. If both true: **downgrade** verdict to `"NEEDS_DISCUSSION"` in the returned object, and log a warning: `"Codex returned APPROVE with N blocking issue(s); downgrading verdict to NEEDS_DISCUSSION."`. `NEEDS_DISCUSSION` is in `VALID_VERDICTS` and is accepted by both plan and code review flows, so a single downgrade target works for both phases.
- Also handle the inverse: `verdict !== "APPROVE" && issues.length === 0`. This is a degenerate response — non-approval with nothing to address. Return `{ valid: false }` so `withRetryOnInvalidResponse` triggers a second attempt rather than the investigator receiving an empty review to respond to.

Downgrade (not invalidate) for the APPROVE+blockers case because it's actionable signal — the issues ARE there — we just don't trust the author's own approval call. Invalidate (retry) for the non-APPROVE+empty case because there's nothing to respond to.

### Normalize escalation recurrence keys

**[Round-4 finding from Codex:]** `issueKey = issue.substring(0, 80)` at `orchestrator.ts:1245` and `:1904` includes the leading `[severity]` prefix. If Codex flips an issue's severity between rounds (malformed `severity` on round 1 → `[unknown]`, correct on round 2 → `[major]`), the same finding gets two different keys and escalation never trips.

Fix: a small helper `normalizeIssueKey(issue: string): string` in the gating module that strips, in order:
1. Leading severity prefix: `^\[(minor|major|critical|unknown)\]\s*`.
2. Leading category prefix (if present): `^\[[A-Z]+\]\s*` — handles CRITICAL / CORRECTNESS / NIT / PRAISE / etc.  Codex could re-phrase the category between rounds ("[CORRECTNESS]" → "[ROBUSTNESS]") while the underlying finding is unchanged; both must key the same.
3. Collapses internal whitespace.
4. Truncates to 80 chars.

Both orchestrator loops use it instead of the raw `substring(0, 80)`.

### Single-source category-to-severity mapping

**[Round-4 finding from Codex:]** The category-to-severity mapping (CRITICAL → critical, CORRECTNESS → major, etc.) currently appears three places in the design: the prompt text via `buildSeverityTaxonomy`, the parser fallback via `inferSeverityFromCategory`, and the gating predicates. Drift between these over time is a real risk.

Fix: the mapping lives as a `const CATEGORY_TO_SEVERITY` record in `src/core/codex-review-gating.ts` (the same new module that hosts `hasBlockingIssues` / `hasOnlyMinorIssues` / `normalizeIssueKey`). All three consumers (`codex-review-prompts.ts` for the prompt table, `codex-reviewer.ts` for the parser fallback, and the gating module itself) import from that one source. Renaming or adding a category requires a single edit.

**Defensive entry for `PRAISE`:** the mapping includes `PRAISE -> minor` even though the prompt instructs Codex to place praise in `summary`, not `issues`. If Codex non-complies and emits a `[PRAISE]` description with a malformed `severity`, the fallback maps it to `minor` instead of `unknown`, which keeps the gating predicates sound (praise is never blocking).

### Harden severity normalization against malformed `severity` fields

**[Round-2 finding from Codex:]** A Codex response with `description: "[CRITICAL] ..."` but a missing/misspelled `severity` field currently parses to `[unknown] ...` at `codex-reviewer.ts:875–884`. Under the new escalation filter (which only counts `[critical|major]`), such issues silently vanish from disagreement tracking even though Codex labeled them critical.

Fix inside `parseStructuredResponse` (still in `codex-reviewer.ts`):

- After extracting the raw `severity`, if it is not one of `minor|major|critical`, scan the `description` for a leading `[CATEGORY]` tag and infer severity from the same mapping table used in the prompt:
  - `[CRITICAL]` → `critical`
  - `[CORRECTNESS|ARCHITECTURE|SECURITY|ROBUSTNESS|COMPLETENESS|ORDERING|RISK]` → `major`
  - `[SUGGESTION|QUESTION|NIT|ALTERNATIVE]` → `minor`
- Only if the description has no recognizable tag either, fall back to the current `unknown`.
- Extract a single helper `inferSeverityFromCategory(description) -> "minor"|"major"|"critical"|null` and reuse it.
- Belt-and-suspenders with the prompt — Codex compliance is the primary defense; this is the safety net.

### Short-circuit the discussion loop when all findings are minor — with full state propagation

**[Round-2 finding from Codex:]** The while-loops at `orchestrator.ts:1228–1232` and `:1887–1891` are purely verdict-driven. Codex could return `NEEDS_DISCUSSION` with only `minor` issues and burn all 5 discussion rounds on items that by definition shouldn't block approval.

**[Round-3 refinement from Codex (CRITICAL):]** Merely exiting the loop early isn't enough. Plan-review approval is recorded at `orchestrator.ts:1347` as `lastPlanApproved = (verdict === "APPROVE")`, and code-review approval at `:2013` as `approved = (verdict === "APPROVE" || isToolFailure)`. The code-review `approved` flag then gates a cycle-continuation at `:587` (`if (!approved && result === "complete") continue`). If we exit the loop with a non-APPROVE verdict, the downstream flags still say "not approved" and force another cycle — which is exactly the round-burn we were trying to avoid.

**Shared predicates (single source of truth):** introduce two helpers in a small shared module (`src/core/codex-review-gating.ts`) so prompt, loop, escalation, and approval all use the same definitions:

```ts
/** Issue starts with `[critical]`, `[major]`, or `[unknown]` — the last because
 *  a dropped/malformed severity could be hiding a real critical/major issue.
 *  Used for escalation counting and loop continuation. */
export function hasBlockingIssues(issues: string[]): boolean;

/** True iff there IS at least one issue AND every issue is `[minor]`.
 *  Empty `issues` returns false — an empty issue list under a non-APPROVE
 *  verdict is ambiguous (could be a model hiccup), not a green light. */
export function hasOnlyMinorIssues(issues: string[]): boolean;
```

**Usage matrix** — every site touches one of these helpers:

| Site | Change |
|---|---|
| plan-review while-loop `:1228–1232` | continue only if `verdict !== APPROVE && hasBlockingIssues(issues)` |
| code-review while-loop `:1887–1891` | same predicate with `reviewRound`/`MAX_CODE_REVIEW_ROUNDS` |
| escalation counter `:1243, :1902` | skip increment for `!hasBlockingIssues([issue])` (i.e. minor/praise) |
| `lastPlanApproved` computation `:1347` | `verdict === "APPROVE" || hasOnlyMinorIssues(issues)` |
| code-review `approved` computation `:2013` | `verdict === "APPROVE" || isToolFailure || hasOnlyMinorIssues(issues)` |

**Three-layer defense (prompt + loop + approval state):**

1. **Prompt rule** in `buildSeverityTaxonomy(...)`: "If every finding is `minor`, your verdict MUST be `APPROVE`. List the minors in `summary` (or a trailing section) — never use `NEEDS_DISCUSSION` / `NEEDS_FIXES` / `MAJOR_CONCERNS` / `MAJOR_PROBLEMS` for a minor-only outcome."
2. **While-loop guard**: exit the loop when there are no blocking issues.
3. **Approval-state guard**: `lastPlanApproved` / code-review `approved` are set by the shared helpers, so a minor-only outcome propagates as approval all the way through `:587` cycle gating.

Log line on short-circuit: `"Plan review had only minor findings (N minor); treating as approved-with-minors and exiting discussion loop."` — distinguishes from a real APPROVE verdict in run logs.

### Make the plan-review MCP paragraph replan-aware

**[Round-2 finding from Codex:]** Dropping the coordinator MCP paragraph is right for the *first cycle* plan review (no tasks, contracts, or decisions yet). For *replans* (cycle 2+), prior cycles have written contracts and decisions — Codex could benefit from knowing about those.

Fix: threadable context on plan-review calls.

```ts
async reviewPlan(planPath: string, round?: RoundBudget, context?: { hasPriorContext?: boolean }): Promise<CodexReviewResult>;
async reReviewPlan(planPath: string, discussionPath: string, round?: RoundBudget, context?: { hasPriorContext?: boolean }): Promise<CodexReviewResult>;
```

**[Round-3 refinement from Codex:]** Gate on `isReplan`, NOT `planVersion > 1`. The `isReplan` parameter already exists in `plan()`'s scope (passed from `orchestrator.ts:361` as `cycleNum > 1`) and is the canonical signal for "have prior cycles produced contracts/decisions?". `planVersion > 1` is almost-but-not-quite equivalent and could drift if the planning loop ever gains a retry path. Pass `{ hasPriorContext: isReplan }` at the call sites.

When `hasPriorContext === true`, include a trimmed MCP paragraph covering only `get_contracts` and `get_decisions` (tasks are still stale mid-cycle and should not be advertised). When `false` (cycle 1), drop the paragraph entirely as originally planned.

Code-review prompts keep the full MCP paragraph unchanged.

### Refactored `codex-reviewer.ts` method signatures

```ts
async reviewPlan(planPath: string, round?: RoundBudget): Promise<CodexReviewResult>;
async reReviewPlan(planPath: string, discussionPath: string, round?: RoundBudget): Promise<CodexReviewResult>;
async reviewCode(taskDescription: string, planPath: string, changedFilesPath: string, diffPath: string, round?: RoundBudget): Promise<CodexReviewResult>;
async reReviewCode(reviewResponsePath: string, changedFilesPath: string, round?: RoundBudget): Promise<CodexReviewResult>;
```

`round` is optional for backwards compatibility with tests that mock the reviewer (e.g. `orchestrator-integration.test.ts`). When absent, the round-budget block is omitted; everything else still applies.

Each prompt is built as:

```
[role setup — 1 line]

[buildAdversarialStance(target)]

[buildFeedbackFraming()]

[buildRoundBudget(round) — only if round is supplied]

[buildSeverityTaxonomy(target)]

[task-specific instructions — what to review, what's attached]

[coordinator MCP server section — always for reviewCode / reReviewCode; trimmed (contracts + decisions only) for reviewPlan / reReviewPlan when hasPriorContext === true; omitted for plan reviews in cycle 1]

[JSON_FORMAT_INSTRUCTIONS — preserved verbatim]
```

**Why the coordinator MCP paragraph is conditional on plan reviews:** at cycle-1 `reviewPlan` / `reReviewPlan` time, no tasks / contracts / decisions exist yet (tasks are materialized at `orchestrator.ts:1382` after plan approval). Telling Codex to call `get_tasks` / `get_contracts` / `get_decisions` would waste a turn returning empty lists. For cycle 2+ replans (`isReplan === true`), prior cycles have already written contracts and decisions, so a **trimmed** paragraph advertising only `get_contracts` + `get_decisions` is included (not `get_tasks`, which is still stale mid-cycle). The MCP server is always wired up — Codex can call it if curious — we just avoid advertising dead tools.

### Orchestrator call-site updates (`src/core/orchestrator.ts`)

Four call sites, two constants. The `RoundBudget.current` threads the **discussion-round number** — i.e. the user-facing "round X of Y" concept, NOT the literal Codex invocation count (which can be 1–2× higher due to the silent retry loop at `codex-reviewer.ts:537 withRetryOnInvalidResponse`). Codex never needs to know about retries.

**Plan review (lines 1221, 1337):**
- softCap = hardCap = `MAX_PLAN_DISCUSSION_ROUNDS + 1` (initial review + up to 5 re-review rounds = 6 discussion rounds).
- Initial `reviewPlan`: `{ current: 1, softCap: 6, hardCap: 6 }`
- Each `reReviewPlan`: `{ current: discussionRound + 1, softCap: 6, hardCap: 6 }` — at the first re-review `discussionRound === 1`, so `current = 2`, etc.

**Code review (lines 1871, 1994):** identical pattern with `reviewRound` and `MAX_CODE_REVIEW_ROUNDS`.

### Prompt size budget

Adding the four blocks adds roughly:
- adversarial stance: ~600 chars
- feedback framing: ~800 chars
- round budget: ~1400 chars
- severity taxonomy: ~1000 chars
- **total overhead: ~3800 chars per prompt, in the base prompt**

The existing aggregate cap `MAX_CODEX_PROMPT_AGGREGATE_CHARS = 2,000,000` applies only to the attached file content in `runCodex` (codex-reviewer.ts:691–763), NOT to the base prompt text. So adding 3800 chars to the base prompt doesn't use up any of the cap's budget — it just grows the final stdin payload by ~0.2% of the cap's size, which is well within Codex's own limits and dwarfed by attached files in every realistic scenario.

## Risks + mitigations

| Risk | Mitigation |
|------|-----------|
| Codex ignores the [CATEGORY] prefix convention | Parser is tolerant — degrades to current behavior |
| Larger prompt hits token limits on very big diffs | Aggregate cap unchanged at 2M chars; new overhead is <0.2% |
| Round counting off-by-one — Codex told it's "round 1 of 6" when it's really round 1 of 6 total invocations | Explicit in the spec: softCap = `MAX_*_ROUNDS + 1` because the discussion-rounds constant excludes the initial review. Verified below. |
| Prompt injection via `taskDescription` in `reviewCode` | `sanitizePromptContent` already applied at line 445; unchanged |
| `reReviewPlan` / `reReviewCode` prompts now assume Codex can re-read the original plan through file attachment | Already true — the plan file is passed as a read path |

### Round-counting verification

From `orchestrator.ts:1221–1337`:

```
reviewResult = reviewPlan(planPath);   // Codex call #1, initial
let discussionRound = 0;
while (discussionRound < MAX_PLAN_DISCUSSION_ROUNDS) {
  discussionRound++;
  // investigator responds (no Codex call)
  reviewResult = reReviewPlan(planPath, responsePath); // Codex call #(discussionRound + 1)
}
```

So total Codex calls in worst case = 1 initial + 5 re-reviews = 6. Matches softCap = 6. ✓

## Open questions (resolved after Codex plan review round 1)

1. `[CATEGORY]` prefix vs. schema change — **resolved**: keep the prefix approach. The [CRITICAL] escalation-counting bug (see fix above) is the real soundness issue, not the encoding.
2. Round-budget block at `current === softCap` — **resolved**: added `FINAL PLANNED ROUND` banner.
3. Feedback framing in re-review prompts — **match ClaudeCodexDiscussion**: include adversarial + framing blocks in every round.
4. Raise MAX_*_ROUNDS — **no**: goal is faster convergence, not more rounds.
5. Coordinator MCP paragraph in plan reviews — **resolved**: omitted for cycle-1 plan reviews (nothing to query), trimmed to contracts+decisions for replan (cycle 2+) plan reviews where prior cycles have written coordination state, and kept in full for code reviews. Gated on `isReplan` at the call sites.
